## 1. 单机运行

适用于开发、测试或 2-4 用户小规模生产场景。所有服务运行在同一台服务器上。

### 1.1 架构

```
单机服务器
├── Nginx (:8000)          ← 统一入口
├── Gunicorn (:8001)       ← 4 workers，FastAPI + SSE
├── Celery Review Worker   ← concurrency=3, queue=review
├── Celery Parser Worker   ← concurrency=2, queue=parser
├── RAG Memory (:3001)     ← 企业知识库检索
├── 本地 workspace/        ← 文件存储
└── 外部 PostgreSQL + Redis
```

### 1.2 前置条件

| 项目 | 要求 |
|------|------|
| 服务器 | 8 核 CPU / 32 GB 内存 / 200 GB SSD |
| 系统 | Ubuntu 22.04 LTS |
| 网络 | 可访问 PostgreSQL、Redis、LLM API |
| Conda | 已安装，py311 环境已创建 |
| Node.js | 18+ |

### 1.3 操作步骤

#### ① 获取代码 & 安装依赖

```bash
cd ~
git clone https://github.com/Zhangdong1977/bjt_agent.git
cd bjt_agent
git submodule update --init --recursive

# 安装后端依赖
conda activate py311
cd backend && pip install -r requirements.txt && cd ..

# 安装前端依赖
cd frontend && npm install && cd ..

# 安装 RAG Memory 依赖
cd rag_memory_service && npm install && cd ..
```

#### ② 配置环境变量

```bash
cd backend
cp .env.example .env
nano .env
```

**必须配置项**：

```bash
# 安全密钥
SECRET_KEY="$(openssl rand -hex 32)"

# 数据库（直连 PostgreSQL）
DATABASE_URL="postgresql+asyncpg://ssirs_user:密码@183.66.37.186:7004/bjt_agent"

# Redis
REDIS_URL="redis://183.66.37.186:7005/0"

# Celery（使用同一个 Redis）
CELERY_BROKER_URL="redis://183.66.37.186:7005/0"
CELERY_RESULT_BACKEND="redis://183.66.37.186:7005/0"

# LLM（三选一）
LLM_PROVIDER="deepseek"
DEEPSEEK_API_KEY="sk-xxx"
DEEPSEEK_API_BASE="https://api.deepseek.com/v1"
DEEPSEEK_MODEL="deepseek-v4-flash"

# 子 Agent 并行数（单机推荐 3）
MAX_SUB_AGENT_CONCURRENCY=3
```

> **注意**：单机模式下 DB_USE_PGBOUNCER 保持默认 false，直连 PostgreSQL。

#### ③ 开发模式启动（调试用）

```bash
cd ~/bjt_agent
conda activate py311
./scripts/bjt.sh start
```

服务清单：
| 服务 | 端口 | 说明 |
|------|------|------|
| Backend API | :8000 | uvicorn 4 workers |
| Frontend | :3000 | vite dev server |
| RAG Memory | :3001 | node dev server |
| Celery Review | - | concurrency=3 |
| Celery Parser | - | concurrency=2 |

#### ④ 生产模式启动

```bash
cd ~/bjt_agent
./scripts/bjt-proc.sh start
```

与开发模式的区别：
- 后端使用 Gunicorn + UvicornWorker（更稳定）
- 前端构建为静态文件，由 Nginx 直接服务
- 后端端口改为 :8001（Nginx 反代）
- 增加了 Celery Beat 调度器

#### ⑤ 配置 Nginx 反代

```bash
sudo cp deploy/nginx/bjt /etc/nginx/sites-available/bjt
sudo ln -sf /etc/nginx/sites-available/bjt /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
```

访问地址：http://服务器IP:8000

#### ⑥ 日常管理命令

```bash
./scripts/bjt.sh status     # 查看所有服务状态
./scripts/bjt.sh stop       # 停止所有服务
./scripts/bjt.sh restart    # 重启所有服务

# 生产模式
./scripts/bjt-proc.sh status
./scripts/bjt-proc.sh logs backend        # 查看后端日志
./scripts/bjt-proc.sh logs celery_review  # 查看审查 Worker 日志
```

#### ⑦ 健康检查

```bash
# API 健康检查
curl http://localhost:8000/health

# Redis 连接检查
python3 -c "import redis; r=redis.Redis(host='183.66.37.186',port=7005); print(r.ping())"

# 队列积压检查
./scripts/task_status.sh
```

## 2. 集群运行

适用于 20 并发生产环境。3 个 API 节点 + 1 个数据库节点 + NFS 共享存储。

### 2.1 架构

```
                    Nginx LB (:8000)
                    (ip_hash 会话粘滞)
                           │
              ┌────────────┼────────────┐
              │            │            │
          Node 1        Node 2        Node 3
        API x1(4w)    API x1(4w)    API x1(4w)
        Review x2      Review x2      Review x3
        Parser x1      Parser x1          │
              │            │            │
              └────────────┴────────────┘
                           │
              PostgreSQL + PgBouncer + Redis
                       NFS 共享存储
```

### 2.2 服务器清单

| 节点 | IP | CPU | 内存 | 角色 |
|------|----|-----|------|------|
| Node 1 | 10.0.0.1 | 8C | 12 GB | API + Review x2 + Parser x1 |
| Node 2 | 10.0.0.2 | 8C | 12 GB | API + Review x2 + Parser x1 |
| Node 3 | 10.0.0.3 | 8C | 16 GB | API + Review x3 |
| DB 节点 | 10.0.0.10 | 8C | 8 GB | PostgreSQL + PgBouncer + Redis |
| NFS | 10.0.0.11 | 2C | 4 GB | 共享文件存储 |
| **合计** | | **26C** | **52 GB** | |

> 以下假设 Node 1-3 的 IP 分别为 10.0.0.1-3，DB 节点为 10.0.0.10。请替换为实际 IP。

### 2.3 操作步骤

#### 第一步：准备数据库节点（10.0.0.10）

**① 安装 Docker & Docker Compose**

```bash
sudo apt update && sudo apt install -y docker.io docker-compose-plugin
sudo systemctl enable --now docker
```

**② 配置 PgBouncer 用户密码**

```bash
# 计算 MD5 哈希（替换 YOUR_PASSWORD 为实际密码）
echo -n "YOUR_PASSWORDssirs_user" | md5sum
# 输出类似: md5a1b2c3d4e5f6...
```

编辑 deploy/pgbouncer/userlist.txt，替换占位符：

```
"ssirs_user" "md5a1b2c3d4e5f6..."
```

```bash
chmod 600 deploy/pgbouncer/userlist.txt
```

**③ 创建 .env 文件**

```bash
cat > .env << 'EOF'
POSTGRES_PASSWORD=YOUR_STRONG_PASSWORD
EOF
```

**④ 启动数据库服务**

```bash
cd ~/bjt_agent
docker compose -f docker-compose-cluster.yml up -d

# 检查状态
docker compose -f docker-compose-cluster.yml ps
```

预期输出：
```
NAME             STATUS       PORTS
bjt-pgbouncer    running      0.0.0.0:6432->6432/tcp
bjt-postgres     running      5432/tcp
bjt-redis        running      0.0.0.0:6379->6379/tcp
```

**⑤ 验证连接**

```bash
# PgBouncer 连接测试
docker exec bjt-pgbouncer psql -h localhost -p 6432 -U ssirs_user -d bjt_agent -c "SELECT 1"

# Redis 连接测试
redis-cli -h 10.0.0.10 -p 6379 ping
```

#### 第二步：准备 NFS 共享存储（10.0.0.11）

> 📌 本节为通用部署模板（IP/网段为占位示例）。**生产实况**（以 [`集群部署方案.md`](集群部署方案.md) §3.9 为准）：NFS server 落在 **node1**（非独立小机也非 prod），实体目录 `/srv/bjt_workspace`，导出经 `/etc/exports.d/bjt_workspace.exports`（不是写进 `/etc/exports`），网段为 `192.168.40.0/24` + `10.1.3.0/24`；三机**统一访问点 `/home/bjt/bjt_agent/workspace`**——node1 本机是软链、node2/3 是 NFS 挂载（`fstab` 选项 `_netdev,auto`）。

```bash
# NFS 服务器上
sudo apt install -y nfs-kernel-server
sudo mkdir -p /srv/bjt_workspace
sudo chown nobody:nogroup /srv/bjt_workspace

# 配置 NFS 导出
echo "/srv/bjt_workspace 10.0.0.0/24(rw,sync,no_subtree_check,no_root_squash)" | \
  sudo tee -a /etc/exports
sudo exportfs -a
sudo systemctl enable --now nfs-kernel-server
```

#### 第三步：准备 API 节点（Node 1/2/3 每台都要执行）

**① 安装基础环境**

```bash
sudo apt update && sudo apt install -y git lsof curl nfs-common

# 安装 Conda + py311（同单机步骤）
# 安装 Node.js 18+（同单机步骤）

# 获取代码
cd ~
git clone https://github.com/Zhangdong1977/bjt_agent.git
cd bjt_agent
git submodule update --init --recursive

# 安装后端依赖
conda activate py311
cd backend && pip install -r requirements.txt && cd ..
```

**② 挂载 NFS 共享存储**

```bash
mkdir -p ~/bjt_agent/workspace

# 挂载 NFS（10.0.0.11 替换为实际 NFS 服务器 IP）
sudo mount -t nfs4 10.0.0.11:/srv/bjt_workspace ~/bjt_agent/workspace

# 验证挂载
df -h ~/bjt_agent/workspace
touch ~/bjt_agent/workspace/.nfs_test && rm ~/bjt_agent/workspace/.nfs_test

# 设置开机自动挂载
echo "10.0.0.11:/srv/bjt_workspace /home/openclaw/bjt_agent/workspace nfs4 defaults 0 0" | \
  sudo tee -a /etc/fstab
```

**③ 配置环境变量**

```bash
cd ~/bjt_agent/backend
cp .env.example .env
nano .env
```

**集群模式关键配置**（与单机不同的部分）：

```bash
# 数据库 -- 连接 PgBouncer（不是直连 PG）
DATABASE_URL="postgresql+asyncpg://ssirs_user:密码@10.0.0.10:6432/bjt_agent"

# PgBouncer 模式开关（必须开启）
DB_USE_PGBOUNCER=true
DB_POOL_SIZE=5
DB_MAX_OVERFLOW=5

# Redis -- 连接集群 Redis
REDIS_URL="redis://10.0.0.10:6379/0"

# Celery -- 使用集群 Redis
CELERY_BROKER_URL="redis://10.0.0.10:6379/0"
CELERY_RESULT_BACKEND="redis://10.0.0.10:6379/0"

# LLM 限流（集群级，所有节点共享同一个 Redis 计数器）
MAX_SUB_AGENT_CONCURRENCY=3

# 其余配置（SECRET_KEY、LLM API Key 等）与单机相同
# !! 所有节点的 SECRET_KEY 必须一致（JWT 跨节点验证）
```

**④ 构建前端**（任选一个节点构建一次即可）

```bash
cd ~/bjt_agent/frontend
npm run build
# 将 dist/ 目录同步到其他节点
scp -r dist/ node2:~/bjt_agent/frontend/
scp -r dist/ node3:~/bjt_agent/frontend/
```

#### 第四步：配置 Nginx LB（在负载均衡节点或 Node 1 上）

编辑 deploy/nginx/bjt-cluster，替换节点 IP：

```nginx
upstream bjt_backend {
    ip_hash;
    server 10.0.0.1:8001 max_fails=3 fail_timeout=30s;
    server 10.0.0.2:8001 max_fails=3 fail_timeout=30s;
    server 10.0.0.3:8001 max_fails=3 fail_timeout=30s;
    keepalive 32;
}
```

部署 Nginx 配置：

```bash
sudo cp deploy/nginx/bjt-cluster /etc/nginx/sites-available/bjt-cluster
sudo ln -sf /etc/nginx/sites-available/bjt-cluster /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx
```

#### 第五步：启动各节点服务

**Node 1**（API + Review x2 + Parser x1）：

```bash
cd ~/bjt_agent
./scripts/bjt-cluster.sh start --node node1
```

**Node 2**（API + Review x2 + Parser x1）：

```bash
cd ~/bjt_agent
./scripts/bjt-cluster.sh start --node node2
```

**Node 3**（API + Review x3，无 Parser）：

```bash
cd ~/bjt_agent
./scripts/bjt-cluster.sh start --node node3
```

> 节点配置文件位于 deploy/node-configs/node{1,2,3}.env，可按需调整 Worker 数量和内存限制。

#### 第六步：验证集群

```bash
# 1. 各节点健康检查
for i in 1 2 3; do
  echo "Node $i:"
  curl -s http://10.0.0.$i:8001/health
  echo ""
done

# 2. LB 健康检查
curl http://10.0.0.1:8000/health

# 3. PgBouncer 连接池状态

# 4. Redis 队列状态
python3 -c "

print(f'Review queue: {r.llen(\"review\")} pending')
print(f'Parser queue: {r.llen(\"parser\")} pending')
"

# 5. NFS 共享验证
# Node 1: echo "test" > ~/bjt_agent/workspace/nfs_verify.txt
# Node 2: cat ~/bjt_agent/workspace/nfs_verify.txt  # 应输出 test
```

### 2.5 监控指标

```bash
# LLM 429 错误计数（按小时，集群级）
python3 -c "
import redis
r = redis.Redis(host='10.0.0.10', port=6379, decode_responses=True)
for k in sorted(r.keys('metrics:llm_429:*')):
    print(f'{k}: {r.get(k)} 次')
"

# LLM 调用指标（按项目）
python3 -c "
import redis, json
r = redis.Redis(host='10.0.0.10', port=6379, decode_responses=True)
for k in r.keys('metrics:llm:*'):
    data = r.get(k)
    if data:
        m = json.loads(data)
        print(f'{k}: calls={m.get(\"call_count\")}, avg_latency={m.get(\"total_latency_ms\",0)//max(m.get(\"call_count\",1),1)}ms')
"

# PgBouncer 连接池使用情况
docker exec bjt-pgbouncer psql -h localhost -p 6432 -U bjt_monitor -d pgbouncer -c "SHOW POOLS;"
```

### 2.6 故障处理

| 场景 | 处理方式 |
|------|---------|
| 单个 API 节点宕机 | Nginx ip_hash 自动剔除；SSE 指数退避重连到其他节点；Celery 任务自动被其他 Worker 接管 |
| Review Worker OOM | max-tasks-per-child=5 自动回收；检查日志 node*_review_*.log |
| LLM API 429 限流 | Redis 分布式限流器自动排队；监控 metrics:llm_429:* 确认频率 |
| NFS 不可用 | 所有节点文件上传/读取失败；检查 NFS 挂载 df -h ~/bjt_agent/workspace |
| PgBouncer 连接耗尽 | SHOW POOLS 查看活跃连接；调大 default_pool_size |
| Redis 宕机 | 所有 SSE/Celery 停止工作；docker restart bjt-redis |

## 快速参考：单机 vs 集群

| 对比项 | 单机 | 集群 |
|--------|------|------|
| 启动命令 | ./scripts/bjt.sh start | ./scripts/bjt-cluster.sh start --node node1 |
| 管理脚本 | bjt.sh / bjt-proc.sh | bjt-cluster.sh |
| 数据库连接 | 直连 PG | 通过 PgBouncer (:6432) |
| DB_USE_PGBOUNCER | false | true |
| DB_POOL_SIZE | 10 | 5 |
| 文件存储 | 本地 workspace/ | NFS 挂载 workspace/ |
| Nginx 配置 | deploy/nginx/bjt | deploy/nginx/bjt-cluster |
| SSE 重连 | 指数退避 + Last-Event-ID | 指数退避 + Last-Event-ID + 节点故障切换 |
| 并发能力 | 2-4 用户 | 20 用户 |
| LLM 限流 | Redis max_concurrent=30 | Redis max_concurrent=30 |
| 429 监控 | Redis metrics:llm_429:* | Redis metrics:llm_429:* |
