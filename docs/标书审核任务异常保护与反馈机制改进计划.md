# 标书审核任务异常保护与反馈机制改进计划

> 分析日期：2026-06-10  
> 范围：用户在项目页点击“立即检查”后，从前端启动、FastAPI 建任务、Celery 执行、MasterAgent/子代理审查、SSE 展示到结果页的完整链路。

## 1. 当前链路概览

用户启动一次标书审核任务后的主链路如下：

1. 前端 `ProjectView.handleStartReview()` 调用 `projectStore.startReview()`。
2. `reviewApi.start()` 请求 `POST /api/projects/{project_id}/review`。
3. 后端 `start_review()` 创建 `ReviewTask(status=pending)`，调用 `run_review.delay(task.id)` 投递 Celery。
4. Celery `run_review()` 将任务置为 `running`，检查项目和已解析的招标/投标文件。
5. `_run_agent_review()` 创建 `MasterAgent`，扫描规则库，生成 `TodoItem`，并发执行 `SubAgentExecutor`。
6. 子代理 `BidReviewAgent` 执行单个规则文件审查，写入 todo 状态、增量保存 `ReviewResult`，通过 Redis Stream 发布 SSE。
7. 前端 `ReviewExecutionView` 通过 `useSSE()` 接收事件，展示主代理、子代理、进度、失败信息，任务完成后跳转结果页。

## 2. 已具备的保护与反馈机制

### 2.1 启动前置保护

- 后端校验项目归属，避免越权启动任务。
- 启动新任务前会把同项目中仍处于 `pending/running` 的旧任务标记为 `failed`，避免多个运行中任务互相干扰。
- 后台任务启动后会检查是否存在已解析的招标文件和投标文件，缺失时写入 `ReviewTask.failed` 并发布 `error` SSE。
- 前端项目页会禁用“立即检查”按钮，直到存在已解析的招标文件和投标文件。

### 2.2 后台执行保护

- Celery 任务有 worker 级软/硬时间限制，审查任务内部也有 `agent_total_timeout` 绝对超时兜底。
- `_progress_watchdog()` 会监控长时间无 SSE 事件的任务，并设置取消信号。
- `BidReviewAgent` 内部有心跳监控，支持 API 取消标记、DB 心跳超时、连续心跳检查失败后的 fail-closed。
- LLM 调用有限流、并发信号量和单次调用超时。
- 子代理异常不会直接中断整个 MasterAgent，单个 todo 会标记失败并通过 `sub_agent_failed` 反馈。
- 子代理支持异常检测和指数退避重试，避免一次空结果或工具失败立即失败。
- 文档路径不存在、MasterAgent 异常、总超时等会被最外层捕获，写入 `ReviewTask.failed` 或 `cancelled`，并发布 `error` SSE。

### 2.3 事件与历史恢复

- SSE 使用 Redis Stream，支持事件短期持久化和 `Last-Event-ID` 断线续传。
- 关键 step 事件会落库到 `AgentStep`，失败/完成任务可通过 API 重新加载历史 todo 和步骤。
- 外部用户的 SSE 会过滤详细 timeline 事件，减少敏感过程信息泄露。

## 3. 主要不足

整体上，系统已经有不少后台保护，但用户可见反馈和状态一致性仍不充分。高风险点如下。

### P0-1：Celery 投递失败会留下不可执行的 pending 任务

位置：`backend/api/review.py` 的 `start_review()` 在创建并 flush `ReviewTask` 后直接调用 `run_review.delay()`。

问题：

- 如果 Redis/Celery broker 短暂不可用，`delay()` 抛异常时，接口会返回 500，但数据库里可能已经存在 `pending` 任务。
- 该任务没有 `celery_task_id`，不会执行，用户只看到“启动审查失败”或再次进入页面看到卡住的 pending。

改进：

- 将 Celery 投递包进 `try/except`。
- 投递失败时把当前任务标记为 `failed`，写入用户可理解的 `error_message`，例如“任务队列暂不可用，请稍后重试”。
- 返回结构化错误码，例如 `REVIEW_QUEUE_UNAVAILABLE`。
- 添加接口测试：模拟 `run_review.delay` 抛异常，断言任务为 failed 且错误可查询。

### P0-2：新执行页未启动 review heartbeat，且启动时存在双 SSE 连接

位置：

- `frontend/src/stores/project.ts` 的 `startReview()` 会调用旧的 `connectSSE()`，旧连接会启动 heartbeat。
- `frontend/src/views/ReviewExecutionView.vue` 挂载后又使用 `useSSE()` 建立新连接，但没有调用 `startHeartbeat()`。

问题：

- 用户从项目页启动后，可能同时存在 store 旧 SSE 和执行页新 SSE。
- 如果执行页刷新或直接进入运行中任务，新页面只连新 SSE，不启动 heartbeat。
- 后端 Agent 现在会主动刷新 `last_heartbeat`，所以不会立即误杀，但“用户关闭页面后自动取消”的产品语义已经不清晰。

改进：

- 统一审查任务实时通道，只保留 `useSSE()` 一套实现。
- 将 heartbeat 生命周期绑定到 `ReviewExecutionView`：运行中任务进入页面时启动，完成/失败/取消/卸载时停止。
- 决策并固化语义：  
  - 若任务应后台持续运行，则移除“前端心跳决定生死”的文案和逻辑，只保留 Agent 自心跳与用户取消。  
  - 若任务应随用户页面关闭而取消，则后端不能再用 Agent 自心跳伪装前端心跳，需要拆分 `frontend_last_seen_at` 和 `worker_last_progress_at`。

### P0-3：子代理全部失败时，整体任务可能仍被标记 completed

位置：

- `MasterAgent._run_sub_agents()` 使用 `asyncio.gather(..., return_exceptions=True)` 收集结果，但只记录日志，不返回成功/失败统计。
- `MasterAgent.run()` 只要未被 cancel，就进入 `_simple_aggregate()` 并返回 `success=True`。
- `_run_agent_review()` 看到 `success=True` 后将 `ReviewTask` 标记为 `completed`。

问题：

- 如果所有规则 todo 都失败，用户仍可能看到整体“已完成”，只是子代理列表里有失败。
- 结果页可能为空，让用户误以为审查通过或无风险。

改进：

- `_run_sub_agents()` 返回统计：`total/completed/failed/cancelled/max_retries_exceeded`。
- 增加整体状态：
  - `completed`：所有必要子代理完成，或失败数低于可接受阈值。
  - `partial_failed`：部分子代理失败，但仍有有效结果。
  - `failed`：全部子代理失败，或关键规则组失败。
- 当前数据库状态如果暂不扩展枚举，可先将“全部失败”标记为 `failed`，将“部分失败”标记为 `completed` 但写入 `warning_message` 或 `metadata.partial_failed=true`。
- 前端必须展示“部分规则审查失败，结果不完整”，并支持用户重试失败子任务。

### P0-4：历史失败任务错误信息可能丢失

位置：`projectStore.selectReviewTask()` 从 `reviewTasks` 列表构造 `currentTask`，但 `ReviewTaskListItem` 不包含 `error_message`，因此失败任务进入执行页时可能显示通用“审查失败”。

问题：

- 后端虽然写了 `ReviewTask.error_message`，但历史列表选择时丢失。
- 用户无法知道失败原因是缺文档、超时、LLM 限流、队列失败还是取消。

改进：

- 后端 `ReviewTaskListItem` 加上 `error_message`。
- 或 `selectReviewTask()` 在选择任务后调用 `getTaskStatus()` 获取完整任务。
- 前端错误展示做错误码映射，不直接裸显英文异常。

### P1-1：SSE 重连耗尽后没有用户可见降级

位置：`useSSE()` 达到最大重连次数后只 `console.error` 并断开。

问题：

- 网络中断或代理层断开时，执行页可能停留在旧进度，没有提示“实时连接已断开”。
- 当前审查任务状态不会自动轮询补偿。

改进：

- `useSSE()` 增加 `onPermanentFailure` 回调。
- 执行页在 SSE 失败后显示状态条：“实时连接已断开，正在轮询任务状态”。
- 启动 `getTaskStatus()` + `getTodosByTask()` 轮询，直到 completed/failed/cancelled。
- 恢复连接成功后停止轮询。

### P1-2：Redis 发布失败只记日志，用户可能收不到错误事件

位置：`_publish_event()` 捕获 Redis 异常后只记录日志，不向任务状态写入可观测告警。

问题：

- 如果 Redis Stream 不可用，任务可能继续执行并落库，但前端完全无进度。
- watchdog 依赖 `_task_last_event_times`，当前在 Redis 写入前就更新时间，可能掩盖“发布失败但任务无用户可见进度”的问题。

改进：

- 将事件发布状态纳入任务诊断字段或独立 `task_events_health`。
- Redis 发布失败连续达到阈值时，写入 `ReviewTask.error_message` 或 `warning_message`。
- watchdog 应区分“内部尝试发布事件”和“成功发布事件”。
- 前端通过轮询状态兜底，不依赖 SSE 必达。

### P1-3：取消反馈不完整

位置：`cancel_review_task()` 对 Celery revoke 异常静默忽略，前端 `RightSidebar.abandonReview()` 失败只写 console。

问题：

- 用户点击“放弃检查”失败时没有可见提示。
- 后端 revoke 失败可能导致任务继续跑一段时间，而前端已切到失败/取消视图。

改进：

- 前端取消失败时显示 message，并保持运行状态。
- 后端取消接口返回 `cancel_signal_set/revoke_requested/revoke_error`。
- 取消后前端继续短轮询任务状态，直到 DB 状态为 `cancelled` 或 `failed`。

### P1-4：错误信息未标准化，排障和用户提示混杂

问题：

- 后端有英文常量、中文超时、原始异常字符串混用。
- 前端大多直接展示 `event.message` 或通用“启动审查失败”。

改进：

- 定义 `TaskErrorCode`：`MISSING_TENDER_DOC`、`MISSING_BID_DOC`、`QUEUE_UNAVAILABLE`、`DOCUMENT_PATH_MISSING`、`LLM_TIMEOUT`、`LLM_RATE_LIMIT`、`AGENT_TOTAL_TIMEOUT`、`SSE_UNAVAILABLE`、`USER_CANCELLED` 等。
- DB 保存 `error_code`、`error_message`、`error_detail`。用户界面展示 message，内部界面可查看 detail。
- SSE `error` 事件也带 `code/retryable/action_hint`。

### P2-1：缺少任务级恢复/补偿机制

问题：

- worker 崩溃后，只有下次启动新任务时才会把旧 running 任务标记 failed。
- 没有周期性 stale task reconciler。

改进：

- 增加后台巡检任务：定期扫描 `pending/running` 且 `updated_at/last_heartbeat` 超阈值的任务。
- 对没有 `celery_task_id`、Celery 状态丢失、worker 已死的任务做失败收敛。
- 给用户提供“重试该任务”入口。

### P2-2：测试覆盖与当前代码存在偏差

观察：

- 部分 heartbeat 测试仍使用旧构造参数 `tender_doc_path/bid_doc_path`，而当前 `BidReviewAgent` 构造函数使用 `tender_docs/bid_docs`。

改进：

- 修正过期测试，避免关键保护逻辑在 CI 中实际未覆盖。
- 增加端到端异常测试矩阵。

## 4. 建议实施计划

### 阶段一：任务状态可靠性（P0）

目标：任何启动失败或全局失败都必须落库、可查询、用户可见。

任务：

1. `start_review()` 增加 Celery 投递失败处理。
2. `ReviewTaskListItem` 或 `selectReviewTask()` 补齐 `error_message`。
3. `MasterAgent._run_sub_agents()` 返回执行统计，并处理“全部失败”。
4. 前端失败页展示真实失败原因和建议动作。

验收：

- Redis/Celery broker 不可用时，启动接口不会留下永久 pending 任务。
- 历史失败任务再次打开能看到具体失败原因。
- 模拟所有子代理失败时，整体任务不再显示为正常完成。

### 阶段二：实时通道与降级（P1）

目标：SSE 不是唯一反馈路径，断线后用户仍能看到任务真实状态。

任务：

1. 移除或停用 store 旧 `connectSSE()` 审查通道，统一使用 `useSSE()`。
2. 明确 heartbeat 语义，并修正前端/后端文案和字段。
3. `useSSE()` 增加永久失败回调。
4. 执行页增加轮询兜底：任务状态、todos、必要时 results。
5. 取消任务增加可见失败提示和取消后状态确认。

验收：

- 断开 SSE 连接后，页面显示降级提示并通过轮询推进状态。
- 恢复网络后不会重复连接或重复事件导致 UI 混乱。
- 取消失败时用户能看到提示，取消成功后状态最终一致。

### 阶段三：错误标准化与运维可观测性（P1/P2）

目标：用户提示、内部排障、运维告警分层清晰。

任务：

1. 增加任务错误码模型和映射表。
2. 后端所有主要异常路径使用 `error_code + user_message + detail`。
3. 记录 SSE 发布失败、DB 状态更新失败、增量保存失败等诊断事件。
4. 增加 stale task reconciler。
5. 补齐异常矩阵测试。

验收：

- 每类异常能在 UI、DB、日志中通过同一个 task_id/error_code 串起来。
- worker 崩溃或 Redis 短暂不可用后，任务最终能自动收敛到明确状态。

## 5. 异常测试矩阵

| 场景 | 期望状态 | 用户反馈 |
|------|----------|----------|
| 缺招标文件 | failed | 提示需上传并解析招标文件 |
| 缺投标文件 | failed | 提示需上传并解析投标文件 |
| 解析记录存在但文件路径丢失 | failed | 提示文档解析结果缺失，请重新解析 |
| Celery broker 不可用 | failed | 提示任务队列暂不可用，可稍后重试 |
| Redis SSE 不可用 | running/completed/failed 由 DB 决定 | 提示实时连接不可用，自动轮询 |
| 单个子代理 LLM 超时 | todo failed 或 retry 后 completed | 展示该规则失败/重试状态 |
| 所有子代理失败 | failed | 提示审查未产生有效结果 |
| 部分子代理失败 | partial/completed with warning | 提示结果不完整，可重试失败规则 |
| 总任务超时 | failed | 提示任务超时终止 |
| 用户取消 | cancelled | 提示已取消 |
| worker 崩溃 | failed by reconciler | 提示后台执行异常，请重试 |

## 6. 结论

当前系统在后台执行层已经具备基础防护：超时、取消、重试、SSE、历史落库、子代理隔离都已存在。但“用户启动任务后的异常保护和反馈”还不能算充足，主要短板集中在任务投递失败、整体状态判定、SSE 降级、历史错误展示和错误标准化。

建议优先完成 P0 项，保证任务状态不会卡死、不会把无效审查误判为完成，并让用户能看到具体失败原因；随后再统一 SSE/heartbeat 架构和轮询兜底。
