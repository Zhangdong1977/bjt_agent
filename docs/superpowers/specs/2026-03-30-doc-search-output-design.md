# 文档搜索工具输出优化设计

## 概述

优化 `DocSearchTool` 在时间线上的输出展示，使其对非技术背景用户更加友好。

## 问题陈述

当前输出存在两个问题：
1. **信息矛盾** - 请求 `full_content: true` 时显示"找到 0 条相关内容"，但实际展示了文档内容
2. **结构不清晰** - 技术术语堆叠，缺乏层次，非技术用户难以快速理解

## 设计目标

1. **问答式输出** - 像助手一样直接回答，而非罗列技术参数
2. **按请求类型区分展示** - keyword 搜索显示匹配条数，full_content 显示文档内容
3. **结构化呈现** - 使用 emoji 标题分类，让非技术用户快速扫描

## 输出格式设计

### 场景1：full_content 请求（获取完整文档）

```
第 X 节
🔍
读取 {tender/bid} 文档
✅ 成功
HH:MM:SS

📄 {招标/投标}书内容摘要

这份{招标/投标}书是关于【项目主题】的文件，主要内容如下：

📌 【分类1标题】
【该分类的内容摘要】

📌 【分类2标题】
【该分类的内容摘要】

...

[完整文档已加载，共 XX 行]
```

**分类策略**：
- 技术要求 → 🛠️ 技术要求
- 工期/时间 → ⏱️ 时间要求
- 预算/价格 → 💰 预算要求
- 资质/证书 → 📋 资质要求
- 其他 → 📝 其他要求

### 场景2：keyword 搜索请求

```
第 X 节
🔍
搜索文档
✅ 成功
HH:MM:SS

🔎 在【招标/投标】书中找到 **N** 处提到"{query}"：

1. 【包含关键词的语句节选】
2. 【包含关键词的语句节选】
3. ...

[或当无结果时]
抱歉，未在【招标/投标】书中找到与"{query}"相关的内容。
```

### 场景3：无关键词请求（仅加载文档）

```
第 X 节
📄
加载{招标/投标}文档
✅ 成功
HH:MM:SS

已加载{招标/投标}书，共 XX 行。
```

## 实现方案

### 修改文件
- `backend/agent/tools/doc_search.py`

### 关键改动

1. **新增 `_extract_summary()` 方法**
   - 从文档内容中提取关键信息分类
   - 使用正则匹配常见分类模式（技术、工期、预算等）

2. **修改 `execute()` 方法的返回格式**
   - 根据请求类型（full_content / query / none）返回不同格式
   - 使用结构化的友好文本替代原始行号罗列

3. **错误消息优化**
   - "No content matching" → "抱歉，未找到相关内容"
   - 保留技术细节在 data 中供调试使用

## 数据流

```
用户请求 (full_content: true)
       ↓
execute() 判断为 full_content 请求
       ↓
_load_document() 加载文档
       ↓
_extract_summary() 提取结构化摘要
       ↓
返回友好格式的 ToolResult
       ↓
前端时间线组件渲染为 UI
```

## 兼容性考虑

- `data` 字段保持不变，包含原始查询结果供调试
- `success` 字段保持不变
- 仅修改 `content` 字段的文本格式

## 测试验证

1. full_content 请求 → 显示文档摘要而非"找到0条"
2. keyword 搜索 → 显示匹配数量和引用
3. 无参数请求 → 显示简洁加载确认
