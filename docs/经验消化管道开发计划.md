## 二、整体架构与阶段划分

### 2.1 消化管道完整数据流

```
┌─────────────────────────────────────────────────────────────────────────┐

│                                                                         │

│                                    │                                    │
│                                    ▼                                    │

│                                    │                                    │

│                                    │                                    │
│                          ┌─────────┼─────────┐                          │
│                          ▼         ▼         ▼                          │
│               ④ MaturityScorer  ⑤ 反馈闭环  ⑥ ExperienceRetriever       │
│               (四轴评分)       (confidence   (检索+相关性校验)            │

### 2.2 阶段划分与依赖关系

```
Phase A (基础层)  ──→  Phase B (提取层)  ──→  Phase C (注入层)  ──→  Phase D (闭环层)
  │                      │                      │                      │
  ├─ 数据模型            ├─ CaseExtractor       ├─ ExperienceRetriever ├─ 反馈管线补完
  ├─ 数据库迁移          ├─ ClusterManager      ├─ Prompt注入          ├─ Skill重写

每个阶段完成后可独立交付验证，即使中途暂停也能产出可衡量的价值。

Prompt 实现要点：

## 四、Phase B：提取层（Case → Cluster → Skill）

### 4.1 目标

实现审核完成后的经验自动提取链路：每次审核产出 Case，Case 经聚类后提取 Skill。本阶段完成后，系统将自动从每次审查中提炼可复用经验。

### 4.2 交付物

#### B1. CaseExtractor

**新建文件**：`backend/experience/case_extractor.py`

**职责**：审核完成后，将 AgentStep 轨迹压缩为一条 `experience_cases` 记录。

**核心流程**：

```
审核完成
  │
  ├─ 获取本次审查的 AgentStep 列表 + ReviewResult 列表
  │
  ├─ CASE_FILTER_PROMPT: 判断是否值得提取
  │   └─ 过滤条件：任务异常终止 + 0 finding + 0 step → 跳过
  │
  ├─ CASE_COMPRESS_PROMPT 阶段1: 工具调用预压缩
  │   └─ 保留语义，丢弃大段返回文本（压缩比 ~10%）
  │
  ├─ CASE_COMPRESS_PROMPT 阶段2: 整体压缩为四元组
  │   └─ (task_intent, approach, key_insight, quality_score_llm)
  │
  ├─ QualityEvaluator.evaluate_findings_batch() → quality_score_eval
  │
  ├─ 融合: quality_score = 0.6 × quality_score_eval + 0.4 × quality_score_llm
  │
  └─ 写入 experience_cases 表
```

**接口设计**：

```python
class CaseExtractor:
    def __init__(self, llm_client=None):
        self._llm_client = llm_client or create_llm_client()

    async def extract(
        self,
        task_id: str,
        project_id: str,
        rule_doc_name: str,
        group_id: str,
        user_id: str,
        agent_steps: list,
        findings: list,
    ) -> dict | None:
        """Extract an experience case from a completed review.

        Returns:
            dict with case fields if extraction succeeds, None if filtered out.
        """
```

**关键实现细节**：
- `task_intent` 模板："审查 {rule_doc_name}，发现 {finding_count} 项{severity_summary}"
- `approach` 必须体现"使用什么工具查了什么关键字→得到了什么→判定依据"
- `quality_score_llm` 评分细则对齐 QualityEvaluator 三维（合规准确性/严重度适当性/完整性）
- AgentStep 数据来源：`backend/models/agent_step.py`
- ReviewResult 数据来源：`backend/models/review_result.py`
- 压缩比目标：~10%（原始上万 token → 几百 token）

#### B2. ClusterManager

**新建文件**：`backend/experience/cluster_manager.py`

**职责**：在线增量聚类，决定新 Case 归属现有簇还是新建簇。

**核心流程**（参照 EverOS `manager.py:404`）：

```
新 Case 进入
  │
  ├─ 阶段1: 嵌入召回（快速过滤）
  │   ├─ 计算 task_intent 的嵌入向量
  │   ├─ 与同 group_id 下现有簇质心做余弦相似度
  │   ├─ 取 top-K 候选簇（K=5）
  │   └─ top-1 相似度 ≥ 0.85 → 直接分配，跳过 LLM
  │
  ├─ 阶段2: LLM 域校验（防止跨域误合并）
  │   ├─ 候选簇的代表 Case + 新 Case → CLUSTER_LLM_ASSIGN_PROMPT
  │   ├─ LLM 判断：主题域 + 问题模式是否同源
  │   └─ 硬约束：禁止跨 group_id 合并
  │
  └─ 输出: cluster_id（现有簇 or 新建簇）
```

**接口设计**：

```python
class ClusterManager:
    def __init__(self, llm_client=None, embedding_service=None):
        self._llm_client = llm_client or create_llm_client()
        self._embedding_service = embedding_service

    async def assign_cluster(
        self,
        case: "ExperienceCase",
        db: "AsyncSession",
    ) -> str:
        """Assign a case to an existing cluster or create a new one.

        Returns:
            cluster_id string
        """
```

**嵌入向量来源**（二选一）：
- 方案一（推荐）：复用 rag_memory_service 的向量化接口
- 方案二：使用 volcengine_embedding_model（已配置在 config.py）

**簇质心维护**：
- 从 `experience_cluster_membership` 表查询同 group_id 下所有簇
- 每个簇的质心 = 该簇所有 Case 的 task_intent 嵌入的均值
- 新 Case 加入后增量更新质心

**配置参数**（参照 EverOS `config.py:32`）：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `similarity_threshold` | 0.65 | 加入现有聚类的最小余弦相似度 |
| `llm_skip_threshold` | 0.85 | Top-1 相似度超过此值时跳过 LLM |
| `llm_top_k_clusters` | 5 | LLM 聚类时 Embedding 预筛选的候选数 |

#### B3. SkillExtractor

**新建文件**：`backend/experience/skill_extractor.py`

**职责**：从 Case 簇中提取或更新 Skill（双模：成功/失败）。

**核心流程**：

```
Case 聚类完成
  │
  ├─ quality_score ≥ 0.5 → 成功路径
  │   ├─ SKILL_SUCCESS_EXTRACT_PROMPT
  │   ├─ 检查簇内是否已有 Skill
  │   │   ├─ 无 → add（新建 Skill, confidence=0.5, skill_form=verified）
  │   │   ├─ 有 + 重叠≥60% → update
  │   │   │   ├─ 新增 Decision 分支 → confidence += 0.1
  │   │   │   ├─ 纯确认 → confidence += 0.05
  │   │   │   ├─ 矛盾 → confidence -= 0.2
  │   │   │   └─ 检测到 ## Potential Steps → 假设晋升, confidence=0.6
  │   │   └─ 有 + 重叠<60% → add（新建 Skill）
  │   └─ 产出: ## Steps + 可选 ## Pitfalls
  │
  └─ quality_score < 0.5 → 失败路径
      ├─ SKILL_FAILURE_EXTRACT_PROMPT
      ├─ 检查簇内是否已有 Skill
      │   ├─ 无 → add（新建 Skill, confidence=0.5, skill_form=hypothesis）
      │   ├─ 有 + verified Skill → 仅追加 Pitfalls, confidence 不变
      │   └─ 有 + hypothesis Skill → 可扩展 Potential Steps + Pitfalls
      └─ 产出: ## Potential Steps + ## Pitfalls（强制）
```

**接口设计**：

```python
class SkillExtractor:
    def __init__(self, llm_client=None):
        self._llm_client = llm_client or create_llm_client()

    async def extract_or_update(
        self,
        case: "ExperienceCase",
        cluster_id: str,
        group_id: str,
        existing_skill: "ExperienceSkill | None",
        db: "AsyncSession",
    ) -> dict:
        """Extract a new skill or update an existing one based on the case.

        Returns:
            dict with action (add/update/none), skill_id, confidence_delta, etc.
        """
```

**置信度锚定规则**（参照设计方案 6.1 节）：

| 事件 | confidence 变化 |
|------|----------------|
| 新建 Skill | = 0.5 |
| 假设晋升 | = 0.6 |
| 新增决策分支 | += 0.1（cap 0.95） |
| 纯确认 | += 0.05（cap 0.95） |
| 矛盾案例（成功 Prompt） | -= 0.2 |
| 矛盾案例（失败 Prompt） | -= 0.15~0.25 |
| 失败案例追加 Pitfall | 不变 |
| 失败案例扩展 Potential Steps | += 0.05 |

#### B4. MaturityScorer

**新建文件**：`backend/experience/maturity_scorer.py`

**职责**：对 Skill 进行四轴评分，决定是否允许注入。

**评分规则**：

| 维度 | 范围 | 低分示例 | 高分示例 |
|------|------|---------|---------|
| Completeness（完整性） | 1-5 → 归一化 0-1 | "检查资质文件"——缺具体步骤 | 完整的发现→验证→定级→建议流程 |
| Executability（可执行性） | 1-5 → 归一化 0-1 | "分析问题"——无具体动作 | "用 DocSearchTool 查询证书过期日→与招标要求对比" |
| Evidence（证据） | 1-5 → 归一化 0-1 | 无任何案例示例 | 多步骤跨场景的真实示例 |
| Clarity（清晰度） | 1-5 → 归一化 0-1 | 一坨散文 | 干净的标题、编号、子项、可扫描 |

**maturity_score** = 四轴均值

**重评分时机**（成本控制）：

| 内容变化比 | 是否重评分 |
|-----------|-----------|
| < 20% | 跳过 |
| ≥ 40% | 强制重评分 |
| 假设晋升 | 强制重评分 |
| 20-40% | 仅当 maturity < 0.6 或 confidence 下降时重评分 |

**核心流程**：

```
BidReviewAgent.run_review() 启动
  │
  ├─ 解析 rule_doc_path → rule_doc_stem (group_id)
  │
  ├─ ExperienceRetriever.retrieve(group_id, query)
  │     │
  │     ├─ SQL 查询 experience_skills
  │     │   WHERE group_id = ?
  │     │     AND retired_at IS NULL
  │     │     AND maturity_score >= 0.6
  │     │     AND confidence >= 0.1
  │     │   ORDER BY maturity_score * confidence DESC
  │     │   LIMIT 10
  │     │
  │     ├─ 向量召回（rag_memory_service hybrid search）
  │     │   query = task_intent
  │     │   filters = {category: experience_skill, group_id}
  │     │   top_k = 10
  │     │
  │     ├─ 合并去重 → 候选 Skill 列表（最多 10 条）
  │     │
  │     └─ LLM 相关性校验 (SKILL_RELEVANCE_VERIFY_PROMPT)
  │         输入：本次审查的招标文件元信息 + 候选 Skill
  │         输出：每条 Skill 的 relevance_score + reason
  │         过滤: relevance_score >= 0.6
  │         排序: maturity × confidence × relevance DESC
  │         截断: top 3
  │
  └─ 返回: list[dict] (最多 3 条)
```

def _render_experience_guidance(self, skills: list[dict]) -> str:
    """Render experience skills into the guidance template."""
    if not skills:
        return ""

    parts = [
        "> 以下经验来自历史审查案例，由系统自动提炼。",
        "> 请参考但不盲从——若与本次实际情况冲突，以本次为准。",
        "",
    ]
    for i, skill in enumerate(skills[:3], 1):
        form_hint = ""
        if skill.get("skill_form") == "hypothesis":
            form_hint = "（来自失败案例，谨慎参考）"
        parts.append(
            f"### 经验 {i}：{skill['name']}"
            f"（置信度 {skill['confidence']:.2f}，"
            f"成熟度 {skill['maturity_score']:.2f}）{form_hint}"
        )
        parts.append(skill.get("description", ""))
        content = skill.get("content", "")
        if len(content) > 150:
            last_period = content[:150].rfind("。")
            if last_period > 50:
                content = content[: last_period + 1]
            else:
                content = content[:150] + "..."
        parts.append(content)
        parts.append("")

    return "\n".join(parts)

async def run_review(self) -> list[dict]:
    # 1. Read rule file

        except Exception as e:

async def _resolve_skill_linkage(db, feedback) -> str | None:
    """Find the most relevant experience_skill for this feedback."""
    from backend.experience.models import ExperienceSkill

    group_id = feedback.rule_doc_name.rsplit(".", 1)[0] if feedback.rule_doc_name else None
    if not group_id:
        return None

    result = await db.execute(
        select(ExperienceSkill).where(
            ExperienceSkill.group_id == group_id,
            ExperienceSkill.retired_at.is_(None),
        ).order_by(
            ExperienceSkill.confidence.desc()

    )
    skill = result.scalar_one_or_none()

async def _apply_confidence_delta(db, skill_id: str, delta: float) -> None:
    from backend.experience.models import ExperienceSkill

    result = await db.execute(
        select(ExperienceSkill).where(ExperienceSkill.id == skill_id)
    )
    skill = result.scalar_one_or_none()

| LLM 输出格式漂移（Skill 提取） | 中 | 中 | 严格 JSON Schema 校验 + 失败重试 + 兜底正则解析 |
| 经验注入占用过多 Context 窗口 | 低 | 中 | 严格限制 3 条/≤600 tokens + 一键关闭开关 |
| 冷启动期效果不明显 | 高 | 低 | 规则文档正例/反例可手工播种为初始 Skill |
| 跨规则文档污染 | 低 | 高 | `group_id` 硬隔离 + Cluster LLM 域校验禁跨 group 合并 |
| 嵌入向量服务不可用 | 中 | 低 | 降级为纯 SQL 查询（跳过向量召回），保证基本可用 |
| 假设晋升误触发 | 低 | 中 | confidence 重置为 0.6（不是更高），强制 maturity 重评分 |
| 模型升级导致经验失效 | 低 | 中 | `experience_cases.metadata` 含模型版本，升级后可按版本批量重置 |
| Skill 提取 LLM 调用成本过高 | 中 | 低 | 重评分时机控制 + 向量召回减少 LLM 调用 + 缓存机制 |
| asyncio.run() 在 Celery prefork 中出错 | 中 | 低 | 已有先例，使用 `max_tasks_per_child=5` 限制 Worker 生命周期 |

## 九、实施优先级建议

如果资源有限，建议按以下优先级逐步实施：

### 优先级 1（必须先完成）

**Phase A 全部** — 基础层是所有后续工作的前提，必须先完成。

### 优先级 2（最小可用，建立质量基线）

**Phase B 中的 B1（CaseExtractor）+ B7（接入审核终点）** — 最小可用组合，先建立质量基线，每次审查自动产出 Case 记录，可以开始追踪审查质量趋势。

### 优先级 3（价值最大）

**Phase C（注入层）** — 即使没有自动 Skill 提取，也可以手工播种 Skill 到 `experience_skills` 表来验证注入效果。这是整个消化管道中价值最大的部分——让 Agent 真正"看到"历史经验。

### 优先级 4（自动化提炼）

**Phase B 中的 B2-B5（聚类+Skill提取）** — 自动化经验提炼，减少人工播种的需求。

### 优先级 5（完整闭环）

**Phase D（闭环层）** — 打通反馈到 Skill 的完整闭环，实现假设晋升和效果验证。

---

这样即使中途暂停，每个阶段都能交付可独立验证的价值：

| 阶段完成后的能力 | 可验证的价值 |
|----------------|------------|
| Phase A | 数据表就绪，配置项可用，Prompt 模板就绪 |
| Phase A + B1 + B7 | 每次审查自动产出 Case，可追踪质量趋势 |
| Phase A + B1 + B7 + C | Agent 能看到手工播种的经验，可 A/B 对比效果 |
| Phase A + B 全部 | 自动从审查中提炼 Skill，无需手工播种 |
| Phase A-D 全部 | 完整自学习闭环：审查→提取→注入→反馈→演进 |
