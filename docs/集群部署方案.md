# 标书审查系统集群部署方案

> 基于 2026-05-26 单任务实测数据，目标为计算支持20并发的系统配置。

## 1. 单机实测分析

### 1.1 架构

```
单机服务器
├── Nginx
├── Uvicorn ×4 (FastAPI + SSE)
├── Celery Review Worker (concurrency=3)
├── Celery Parser Worker (concurrency=2)
├── 本地文件存储 workspace/
└── 外部 PostgreSQL + Redis
```

### 1.2 单任务资源消耗（实测）

| 资源 | 实测值 |
|------|--------|
| Review 内存 | ~630MB（75MB 空闲 → 632MB 峰值） |
| Parser 内存 | ~640MB（75MB 空闲 → 642MB，20s 内） |
| CPU | avg 25.5%, max 78.8% |
| LLM 调用次数 | 658 次/任务 |
| LLM 平均延迟 | 5.2s（范围 1.3s ~ 54.8s） |
| LLM 总 Tokens | 20.7M（平均 31,508/次） |
| 单任务耗时 | ~48 min |
| DB 连接增量 | +6（空闲 12 → 峰值 18，活跃仅 1） |
| Redis 内存增量 | 3MB → 8.2MB（608 Keys） |

### 1.3 单机结论

单机配置可支撑 2-4 用户。扩展至 20 并发需：
- 7 个 Review Worker（×3 并发 = 21 槽位）
- 分布式 LLM 限流
- 共享文件存储
- PgBouncer 连接池

## 2. 20 用户并发配置

### 2.1 部署拓扑

```
                    Nginx LB (ip_hash)
                           │
              ┌────────────┼────────────┐
              │            │            │
          Node 1        Node 2        Node 3
        API×1(4w)     API×1(4w)     API×1(4w)
        Review×2      Review×2      Review×3
        Parser×1      Parser×1           │
              │            │            │
              └────────────┴────────────┘
                           │
              PostgreSQL + PgBouncer + Redis
                       NFS 共享存储
```

### 2.2 节点配置

| 节点 | 服务 | CPU | 内存 |
|------|------|-----|------|
| Node 1 | API(4w) + Review(c=3)×2 + Parser(c=2)×1 | 8C | 12GB |
| Node 2 | API(4w) + Review(c=3)×2 + Parser(c=2)×1 | 8C | 12GB |
| Node 3 | API(4w) + Review(c=3)×3 | 8C | 16GB |
| DB 节点 | PostgreSQL + PgBouncer + Redis | 8C | 8GB |
| NFS | 共享存储 | 2C | 4GB |
| **合计** | | **22C** | **52GB** |

### 2.3 并发能力

| 指标 | 容量 | 计算依据 |
|------|------|---------|
| 审查并发 | 21（7 Worker × 3） | 目标 20，留 1 余量 |
| 解析并发 | 4（2 Worker × 2） | 解析占比低，4 并发够用 |
| SSE 连接 | ~200 | 3 API × 4 workers，SSE 为主 |
| DB 连接 | ~216（PgBouncer 降至 50-80） | 21 任务 × 6 + 3 API × 30 |

### 2.4 Worker 关键参数

| 参数 | 值 | 依据 |
|------|-----|------|
| `--concurrency` | 3 | 3 任务 × 630MB ≈ 1.9GB/Worker |
| `--max-memory-per-child` | 2500000 (2.5GB) | 1.9GB + 600MB 余量 |
| `--max-tasks-per-child` | 5 | 实测内存不主动释放，定期回收 |
| `MAX_SUB_AGENT_CONCURRENCY` | 3 | 任务内子 Agent 并行数 |
| LLM 限流 `max_concurrent` | 30（集群级） | 基于 Redis INCR/DECR 分布式限流 |

### 2.5 计算理由

**Worker 配置：**
- 单任务 Review 内存 630MB，3 并发 = 1.9GB，加主进程 ≈ 2GB
- 单任务 Parser 内存 640MB，2 并发 = 1.3GB，加主进程 ≈ 1.5GB
- Node 1/2: 2×2GB(Review) + 1.5GB(Parser) + 0.5GB(API) + 3GB(系统) ≈ 9GB → 配 12GB
- Node 3: 3×2GB(Review) + 0.5GB(API) + 3GB(系统) ≈ 9.5GB → 配 16GB

**LLM 并发分析：**
- 21 任务 × 3 子 Agent 并行 = 63 瞬时请求
- 限流器 max_concurrent=30，平均排队延迟 = (63-30)/30 × 5.2s ≈ 5.7s
- 单任务 658 次调用 ÷ 48min ≈ 0.23 QPS，20 并发 ≈ 4.6 QPS（集群平均）
- 需确认 MiniMax API RPM/TPM 限额是否满足

**DB 连接：**
- 单任务实测增量 +6 连接，21 任务 ≈ 126 连接
- API 3 实例 × 30 = 90，总计 ~216
- PgBouncer transaction 模式压缩至 50-80 实际连接

**Nginx SSE 要点：**
- `ip_hash` 会话粘滞
- `proxy_buffering off`、`proxy_read_timeout 86400`
- 前端指数退避重连 + `last_event_id` 恢复，节点故障自动切换

### 2.7 主要风险

| 风险 | 应对 |
|------|------|
| LLM API 限流（最关键） | Redis 分布式限流器 max_concurrent=30，监控 minimax返回 429 错误，建议更新为预付费deepseek服务+本地GPU服务器的OCR图片识别 |
| Worker 内存泄漏 | `max-tasks-per-child=5` + `max-memory-per-child=2.5GB` |
| SSE 连接漂移 | 前端指数退避重连 + Redis Stream 保留 1 小时 |
| Parser 大文档内存 | `max-memory-per-child=2GB`，大内存节点部署 |

## 3. 实际部署实例（2026-07 扩容 · 已实测）

> 2026-07-01 落地本节。前 §1–2 是设计基线（理论 10.0.0.x），本节是**实测后的真实拓扑与可执行方案**，二者有差异处以本节为准。通用安装步骤仍见 [运行操作指导.md](运行操作指导.md)，本节只写**实例化的 IP、决策、调整点、切换流程**。
>
> 🔒 本文件入库（GitHub），**只写 IP 不写密码**；真实凭证在 `sales/doc/workspace/05-production-env.md`（非 git 受控散落文件）。

### 3.1 实测网络拓扑（4 机全互通，已验证）

4 台机器在**同一台 NAT/防火墙设备**后面，共享公网出口 IP `120.194.14.251`，内网分两个子网，防火墙已开启子网间路由：

```
                         公网
                check.aibjt.com:30002  (云防火墙 30002 → 内网 nginx:8000)
                           │
        ┌──────────────────┴───────────────────┐
        │  prod  120.194.14.251  (ens160 10.1.3.241/24, gw 10.1.3.254)
        │   ▲ 边缘 + DB 枢纽（详见 3.3）
        │   - nginx-LB (:8000, SSL 在现网既有监听上终止)
        │   - PostgreSQL  bjt-postgres 容器 (bjt_db / bjt_user)   ← 复用，不新建
        │   - PgBouncer (:6432)   ← 新增，指到上面 PG
        │   - Redis (:6379)       ← 现有，重绑内网 + 加密码
        │   - NFS server  导出 /srv/bjt_workspace
        │   - RAG Memory (:3001) + celery beat                  ← 单例服务留在枢纽
        └───────┬────────────────┬────────────────┬──────────────┘
            子网间路由        子网间路由        子网间路由      （同一 NAT，已实测互通）
                ▼                ▼                ▼
        node110            node111           node112
        192.168.40.110     192.168.40.111    192.168.40.112
        (ens160 /24, gw 192.168.40.254)
        API(4w)+Rev×2+Par×1   API(4w)+Rev×2+Par×1   API(4w)+Rev×3
        NFS 挂载              NFS 挂载              NFS 挂载
```

**已实测的可达性矩阵**（`OK`=通 / `REFUSED`=路由通但端口未监听 / `TIMEOUT`=不通）：

| 从 → 到 | :22 | :8001 | :30001 | :30002 | :6432/6379/2049 |
|---|---|---|---|---|---|
| prod → node110/111/112 | OK | REFUSED¹ | – | – | – |
| node110/111/112 → prod | – | – | OK | OK | REFUSED² |
| node → node | OK | – | – | – | – |

¹ `:8001` REFUSED = 节点还没部署 backend，符合预期；部署后即通。
² `:6432/6379/2049` REFUSED = **路由已通**，只是 prod 上 PgBouncer 未起、Redis/NFS 还绑在回环。把服务绑到 `10.1.3.241`（或 `0.0.0.0` 内网）即通——**不需要找网管开端口**。

外联：4 台均可达 `api.deepseek.com:443`、`aip.baidubce.com:443`（百度 OCR）、`github.com:443`（全部 OK）。

### 3.4 与原设计（§2）的差异点（重要）

| 项 | 原设计 §2 | 实测实例（本节） | 原因 |
|---|---|---|---|
| DB 节点 | 新起一台跑 `docker-compose-cluster.yml`（全新 PG `bjt_agent`/`ssirs_user`）| **复用 prod 现有 `bjt-postgres`（`bjt_db`/`bjt_user`），仅加 PgBouncer** | 现网已有数据（review_tasks / ai_usage_* / experience 技能），避免迁移风险 |
| NFS 节点 | 独立小机（10.0.0.11）| **NFS server 就放在 prod** | 4 机同 NAT，prod 是天然中心；省一台 |
| LB 节点 | 独立或 node1 | **prod 既是公网入口也是 LB** | prod 已持有公网 IP / TrustAsia 证书 / DNS / 云防火墙 30002 映射，不动最稳 |

| 单例服务 | 未明确 | **RAG(:3001) + celery beat 留在 prod** | 单例不宜散到 worker 节点；prod 已有现成环境 |

### 3.5 既有资产需调整清单（部署前先改这些）

> 全部在 `sales/bjt-agent` worktree 改，合到 `master` 再 `git pull` 到各机。

1. **`deploy/pgbouncer/pgbouncer.ini`**——`[databases]` 段由占位改为指向 prod 现有库：
   ```ini
   [databases]
   bjt_db = host=127.0.0.1 port=5432 dbname=bjt_db
   ```
   `auth_type = md5`，`auth_file` 指向 `userlist.txt`，里面放 `bjt_user` 的 MD5（`echo -n "PWDbjt_user" | md5sum`）。

2. **`docker-compose-cluster.yml`**——**不要**用它起全新 PG。改为只起 PgBouncer 一个服务，`DATABASE_URL` 指到现有 `bjt-postgres`（host=宿主，port=5432，user=bjt_user，db=bjt_db）。或干脆 `apt install pgbouncer` 裸装，免走 docker。**Redis 也不要用它起**——见第 3 项"旁路 Redis"。

3. **旁路 Redis**（零中断关键，prod）：现网 `:6379` 绑回环、无密码，**全程不动它**（动它 = 重启 = 打断现网 celery，是切换里唯一的硬中断点）。改为在 prod **新起一个独立 Redis 实例 `:6380`**：`bind 10.1.3.241`、`requirepass <REDIS_CLUSTER_PWD>`、独立数据目录与持久化。集群节点全连 `:6380`，现网单机继续用 `:6379`，互不影响；切换后 prod 计算下线，`:6379` 自然排空退役，最终集群只留 `:6380`（届时也可把它规范回标准 `:6379`）。
   - **备选（需低峰窗，省一个实例）**：直接给现网 `:6379` 加 `bind 10.1.3.241 127.0.0.1`（**暂不加密码**）重启一次——Redis 重启约 2–5s，prod 后端自动重连（未改密码 → `.env` 不动 → 不掉线），属"轻微抖动"非宕机；集群跑稳后再加 `requirepass` 并**滚动重启节点**（节点在 LB 后，滚动 = 零中断）。能接受这点抖动就走这条。下文 runbook（§3.6）按"旁路 `:6380`"主路径写。

4. **`deploy/nginx/bjt-cluster`**——`upstream` 改为实测 IP（**保留 prod 现有 SSL 监听**，只换 upstream）：
   ```nginx
   upstream bjt_backend {
       ip_hash;
       server 192.168.40.110:8001 max_fails=3 fail_timeout=30s;
       server 192.168.40.111:8001 max_fails=3 fail_timeout=30s;
       server 192.168.40.112:8001 max_fails=3 fail_timeout=30s;
       keepalive 32;
   }
   ```
   ⚠️ prod 现网 nginx 已在 `:30002`(ssl) 终止 SSL 并反代到 `127.0.0.1:8001`。**不要整体替换为 `bjt-cluster`（它 `listen 8000` 无 ssl）**，而是把上面 `upstream bjt_backend` 合进 prod 现有 site 配置，把 `proxy_pass` 从 `http://127.0.0.1:8001` 改成 `http://bjt_backend`，其余 SSL/证书/header/SSE 配置不动。

5. **`scripts/bjt-cluster.sh` 缺 celery beat**——该脚本只起 review/parser worker + backend + rag，**不起 beat**。beat 是单例调度器，全集群只能起一个。**方案**：在 prod 上用 `bjt` 用户单独起 `celery -A celery_app beat`（走 conda py311），不放进 `bjt-cluster.sh`。或在 `bjt-cluster.sh` 加一个 `START_BEAT` 开关，仅在 node1 启用。

6. **节点 `backend/.env`（集群版，每台 node 一份，关键差异项）**：
   ```bash
   # 复用 prod 库，走 PgBouncer
   DATABASE_URL=postgresql+asyncpg://bjt_user:<BJT_USER_PWD>@10.1.3.241:6432/bjt_db
   DB_USE_PGBOUNCER=true
   DB_POOL_SIZE=5
   DB_MAX_OVERFLOW=5
   # 集群 Redis（旁路实例 :6380，带密码；现网 :6379 不动）
   REDIS_URL=redis://:<REDIS_CLUSTER_PWD>@10.1.3.241:6380/0
   CELERY_BROKER_URL=redis://:<REDIS_CLUSTER_PWD>@10.1.3.241:6380/0
   CELERY_RESULT_BACKEND=redis://:<REDIS_CLUSTER_PWD>@10.1.3.241:6380/0
   # RAG 单例在 prod
   RAG_MEMORY_SERVICE_URL=http://10.1.3.241:3001
   # JWT 跨节点校验——三台 node 必须一致，且与 prod beat 端一致
   SECRET_KEY=<同一个值>
   # LLM = DeepSeek，图像理解 = 百度 OCR
   LLM_PROVIDER=deepseek
   DEEPSEEK_API_KEY=<sk-xxx>
   IMAGE_UNDERSTANDING_PROVIDER=baidu
   BAIDU_OCR_APP_ID=<...>
   BAIDU_OCR_API_KEY=<...>
   BAIDU_OCR_SECRET_KEY=<...>
   MAX_SUB_AGENT_CONCURRENCY=3
   ```
   > `OPERATE_API_BASE_URL` / `USAGE_SYNC_API_KEY` 等与现网单机保持一致（见 `05-production-env.md`）。

### 3.6 零/极小中断切换 runbook（旁路搭建 → 单次 reload 切换 → 排空下线）

**核心思路**：现网 prod 全程保持对外服务**不变**；集群在**旁路**并行搭建并端到端验证；用户流量的切换 = 一次 `nginx -s reload`（对新建连接零中断，既有 SSE 连接在 prod 上自然排空）；prod 在途任务排空后，才下线其计算角色（此时流量已在节点，对用户不可见）。**全程无硬中断**，唯一边界见下方"在途任务连续性"。

**中断预算**：

| 步骤 | 中断量 | 说明 |
|---|---|---|
| 阶段 0 旁路搭建（起 PgBouncer / 旁路 Redis / NFS rsync）| **0** | 全是增量，不碰 prod 现有配置与进程 |
| 阶段 1 端到端验证 | **0** | 公网流量仍全进 prod，node 只在内部自测 |
| 阶段 2/3 `nginx -s reload` 切换 | **0** | reload 平滑；ip_hash 让既有用户留 prod 直至自然重连 |
| 阶段 4 prod 排空 + 停计算 | **0** | 流量已在节点，停 prod gunicorn/celery 对外不可见 |
| （仅"备选"路径）现网 Redis 加 bind 重启 | **2–5s 抖动** | 选旁路 `:6380` 即无此项；低峰窗可忽略 |

**在途任务连续性（唯一边界）**：切换瞬间，已在 prod 跑的审查任务继续在 prod 跑完（其 celery/SSE/文件都在 prod）。因 `ip_hash` 粘滞，发起该任务的用户不重连就一直在 prod 看到进度；若中途重连，可能被分到某 node（node 的旁路 Redis 没有该任务事件）→ 时间线暂停，刷新或等任务在 prod 跑完即恢复。**规避**：切换选低峰、在途任务少的时段；切换后等 prod celery 队列排空再下线（阶段 4）。

---

**阶段 0｜旁路搭建（prod 完全不动，0 中断）**

0.1 三台 node 装基础：`conda(py311)` + `node18` + `nfs-common` + `git`；`git clone` bjt_agent + `git submodule update --init Mini-Agent` + `pip install -r backend/requirements.txt`。与 prod 无交互。

0.2 prod 起 **PgBouncer**（apt 或 docker），监听 `0.0.0.0:6432`，`[databases] bjt_db=host=127.0.0.1 port=5432 dbname=bjt_db`，userlist 用 `bjt_user` 的 MD5。**prod 自己的后端仍直连 `:5432`，不改**。从 node 验证：`psql -h 10.1.3.241 -p 6432 -U bjt_user -d bjt_db -c 'SELECT 1'`。

0.3 prod 起**旁路 Redis** `:6380`（§3.5 第 3 项）。现网 `:6379` **不动**。

0.4 prod 装 `nfs-kernel-server`，导出**新目录** `/srv/bjt_workspace` 给 `192.168.40.0/24 10.1.3.0/24`。**`rsync -aH` 复制**（不是移动）现网 `/home/bjt/bjt_agent/workspace/` → `/srv/bjt_workspace/`；prod 继续用旧路径读写。三台 node：`mount -t nfs4 10.1.3.241:/srv/bjt_workspace ~/bjt_agent/workspace` + 写 `/etc/fstab` + 跨机写读验证。
> 切换前可再做一次增量 `rsync` 兜底（知识库/历史文件一致）；新任务切换后直接写 NFS。

0.5 三台 node 配 `backend/.env`（§3.5 第 6 项），关键：DB 走 `10.1.3.241:6432`、Redis/Celery 走 `10.1.3.241:6380`（旁路，带密码）、`RAG_MEMORY_SERVICE_URL=http://10.1.3.241:3001`、**`SECRET_KEY` 复制 prod 现值**（切换前后 JWT 连续、用户不掉线）、`LLM_PROVIDER=deepseek`、`IMAGE_UNDERSTANDING_PROVIDER=baidu`。
