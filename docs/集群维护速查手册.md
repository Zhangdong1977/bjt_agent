# 集群维护速查手册（day-2 ops）

> 集群上线后的**日常维护 / 排障 / 扩缩容**速查。面向"出了状况要快速处置"或"要改点东西"的场景，命令以可直接复制为主。
> 部署与设计见 [集群部署方案.md](集群部署方案.md)（§3 实际部署实例、§3.9 落地实况）；单机/集群起停见 [运行操作指导.md](运行操作指导.md)。
> 🔒 本文件入库（GitHub），**不含密码**；真实凭证见 `sales/doc/workspace/05-production-env.md`。

**约定**：除非另注，命令**在 prod（`120.194.14.251`）上执行**——prod 已配好到各节点的 SSH 别名 `bjt-node1`/`bjt-node2`/`bjt-node3`（见 `sales/scripts/cluster-deploy/10_ssh_mesh.py`）。从公网登录 prod：`ssh -p 30001 root@120.194.14.251`。节点是私网 `192.168.40.110/111/112`，公网不可达——要么在内网/VPN 直连，要么 `ssh -J root@120.194.14.251 -p 30001 root@192.168.40.110` 经 prod 跳。

## 0. 拓扑与端口速查

| 角色 | 机器 | 关键端口 |
|---|---|---|
| 边缘 + DB 枢纽 | prod `120.194.14.251`（内网 `10.1.3.241`）| nginx `:8000`(ssl, 公网经云防火墙 30002→8000)、PG `:5432`(docker)、Redis `:6379`(docker)、PgBouncer `:6432`(备用)、RAG `:3001` |
| 计算节点 | node1 `192.168.40.110` / node2 `.111` / node3 `.112` | gunicorn `:8001`（在 nginx upstream `bjt_backend` 里）|
| 共享存储 | node1 `/srv/bjt_workspace`（NFS server）→ node2/node3 挂载到 `/home/bjt/bjt_agent/workspace` | NFS `:2049`（仅 node1）|

数据面：节点 → `10.1.3.241:5432`(PG 直连) + `10.1.3.241:6379`(Redis 共用)。Redis 分库：**db0 = 业务缓存 + SSE Stream(`sse:stream:<session>`)**、**db1 = celery broker(review/parser 队列)**、**db2 = celery result backend**。**Redis/PG 是全集群共享的单一状态源**，跨节点自洽。看任务进度条数：`docker exec bjt-redis redis-cli -n 0 XLEN sse:stream:<session_id>`。

## 1. 日常巡检（一键）

```bash
# 在 prod 上跑：3 节点 health + celery + 队列 + PG 连接 + nginx 5xx
for n in node1 node2 node3; do
  echo "== $n =="
  ssh bjt-$n "curl -s http://127.0.0.1:8001/health; echo; \
              pgrep -af 'celery.*-Q review' | wc -l | xargs echo 'review_pids:'; \
              pgrep -af 'celery.*-Q parser' | wc -l | xargs echo 'parser_pids:'"
done
echo "== 队列 =="; docker exec bjt-redis redis-cli -n 1 LLEN review; docker exec bjt-redis redis-cli -n 1 LLEN parser
echo "== PG 连接 =="; docker exec bjt-postgres psql -U bjt_user -d bjt_db -tAc "select count(*) from pg_stat_activity"
echo "== nginx 近10分钟 5xx =="; awk -v d=$(date -d '-10 min' +%d/%b/%Y:%H:%M) '$0~d && / 5[0-9][0-9] /' /var/log/nginx/access.log | wc -l
```

健康判据：3 节点都 `{"status":"healthy"}`；review_pids 每节点 ≥8（node3 ≥12）；队列不持续堆积；PG 连接 < 80；5xx = 0。

## 2. 服务起停

### 节点（node1/2/3）— 用 `bjt-cluster.sh`
⚠️ **必须先 source conda**（celery worker 继承调用者 shell 环境，否则只有 backend 起得来）：
```bash
# 以 node1 为例（在 prod 上经 mesh，或登 node1 本机）
ssh bjt-node1 "su - bjt -c '
  source /home/bjt/miniconda3/etc/profile.d/conda.sh && conda activate py311 && \
  cd /home/bjt/bjt_agent && \
  ./scripts/bjt-cluster.sh status --node node1        # 查状态
  ./scripts/bjt-cluster.sh stop  --node node1          # 停
  ./scripts/bjt-cluster.sh start --node node1          # 起
'"
```
> `restart` 实现是 stop+start，可行；但 stop 不会等 celery 优雅排空，**有在途任务时优先 stop→等队列空→start**。节点 stop 期间 nginx 自动把它摘掉（`max_fails=3 fail_timeout=30s`），用户请求路由到其余节点，不中断。

### prod 单例服务（RAG / celery beat）— 用 `bjt-proc.sh`
```bash
ssh -p 30001 root@120.194.14.251   # 登 prod，必须交互式 login shell
su - bjt                            # 进 bjt（.bashrc 激活 conda+nvm）
cd /home/bjt/bjt_agent
./scripts/bjt-proc.sh status        # 看 backend/celery×3/rag_memory
./scripts/bjt-proc.sh stop && ./scripts/bjt-proc.sh start
```
> Phase 4 之前 prod 仍跑 gunicorn+celery（热备，**不在 nginx upstream**）+ RAG + beat。**beat 全集群只能这一个**，别在节点上另起。

### DB / Redis / PgBouncer（prod，systemd/docker）
```bash
docker restart bjt-postgres         # PG（慎：全集群断）
docker restart bjt-redis            # Redis（慎：celery/SSE 全断）
systemctl restart pgbouncer         # PgBouncer（备用，节点没用到不影响）
nginx -t && systemctl reload nginx  # nginx 改配置后
```

## 3. 代码与配置更新

**节点不是 git 仓库**（从 prod rsync 而来）。更新流程：

```bash
# 1) 先把新代码弄到 prod（prod 是 git 仓库，master 分支）
ssh -p 30001 root@120.194.14.251 "su - bjt -c 'cd /home/bjt/bjt_agent && git pull --ff-only origin master'"
#   ⚠️ backend/models/ 被 gitignore，新模型文件要 git add -f（见 CLAUDE.md）

# 2) 把改动的文件 rsync 到 3 节点（全量 backend 也可）
for n in node1 node2 node3; do
  rsync -aH --chown=bjt:bjt --exclude='__pycache__' --exclude='*.pyc' \
    /home/bjt/bjt_agent/backend/  bjt-$n:/home/bjt/bjt_agent/backend/
done
#   有 schema 变更：先 docker exec bjt-postgres psql ... < backend/migrations/NNN_xxx.sql
#   （init_db 的 create_all 只建新表，不能 ALTER 已有表）

# 3) 重启受影响节点的服务（backend 改→重启 backend，celery 任务逻辑改→重启 celery）
for n in node1 node2 node3; do ssh bjt-$n "su - bjt -c 'source ~/miniconda3/etc/profile.d/conda.sh && conda activate py311 && cd /home/bjt/bjt_agent && ./scripts/bjt-cluster.sh stop --node $n && ./scripts/bjt-cluster.sh start --node $n'"; done
```

**`.env` 改动**：`.env` 在每台节点本地（不在 git、不随 rsync，因为部署时被排除）。改一处后要**手工同步到全部 3 节点** + 重启该节点的 backend（celery 不读大部分配置）。prod 的 `.env` 与节点**不同**（DB/Redis host 不同），别照抄。

**worker 数量**：改 `deploy/node-configs/nodeN.env`（`REVIEW_WORKER_COUNT`/`REVIEW_CONCURRENCY` 等），rsync 到节点 → 重启该节点。

## 4. 扩缩容

| 想做什么 | 怎么做 |
|---|---|
| 加/减某节点 review worker | 改 `deploy/node-configs/nodeN.env` 的 `REVIEW_WORKER_COUNT`，rsync 到该节点，`bjt-cluster.sh restart --node nodeN` |
| 临时下掉一个节点 | `bjt-cluster.sh stop --node nodeN`（nginx 30s 内摘除，其余节点承接）|
| 加第 4 个节点 | 复用 `sales/scripts/cluster-deploy/` 的 `10→20` 脚本（建 SSH mesh、装环境、rsync 代码、配 .env、起服务），然后把新节点 IP 加进 prod nginx 的 `upstream bjt_backend` 并 reload |
| 切 PgBouncer（连接数吃紧时）| 节点 `.env`：`DATABASE_URL` 端口 `5432→6432`、`DB_USE_PGBOUNCER=true`；**前提**：合并 worktree 里 `review_tasks.py`/`document_parser.py` 的 `statement_cache_size=0` 补丁到 master 并 rsync 到节点。逐节点重启（滚动，零中断）|

## 5. 日志速查表

| 看什么 | 在哪 |
|---|---|
| **审查任务执行**（最常用）| 节点 `/home/bjt/bjt_agent/scripts/logs/node{1,2,3}_review_1.log`、`_review_2.log`（celery worker 全生命周期 + 应用 INFO/ERROR，含 `[PARSE]`/任务进度）|
| 解析任务执行 | 节点 `.../nodeN_parser_1.log` |
| backend API 启动/uvicorn | 节点 `.../nodeN_backend.log`（stdout）、`nodeN_backend_error.log`（gunicorn 错误）|
| celery beat / RAG | prod `/home/bjt/bjt_agent/scripts/logs/celery_beat.log`、`rag_memory.log` |
| nginx（用户请求）| prod `/var/log/nginx/access.log`、`error.log` |
| PostgreSQL | prod `docker logs bjt-postgres` |
| Redis | prod `docker logs bjt-redis` |
| PgBouncer | prod `/var/log/postgresql/pgbouncer.log` |

> ⚠️ **HTTP 访问行不记录**：gunicorn 用 `UvicornWorker`，`--access-logfile` 不生效（prod 同款）。要 HTTP 访问日志，后续给 gunicorn 加 `--access-logformat` 或 uvicorn `--access-log`。

实时追某个节点的审查日志：`ssh bjt-node1 tail -f /home/bjt/bjt_agent/scripts/logs/node1_review_1.log`

## 6. 常见故障与处理

| 现象 | 排查/处理 |
|---|---|
| 某节点宕机 | nginx `max_fails=3` 30s 内自动摘除；celery 在途任务被其它 worker（含 prod 热备）接管；SSE 客户端指数退避重连到存活节点。修好后 `bjt-cluster.sh start` 重新入组 |
| 审查 worker OOM | `--max-memory-per-child=2500000`(2.5G) 自动回收子进程；查 `node3_review_*.log`（node3 内存最紧 11G）；必要时把 node3 的 `REVIEW_WORKER_COUNT` 降到 2 |
| PG 连接耗尽（`max_connections=100`）| `select client_addr,count(*) from pg_stat_activity group by 1;` 看谁占；切 PgBouncer（见 §4）；或临时调大 `docker exec bjt-postgres psql -c "alter system set max_connections=200"` + 重启 PG |
| Redis 挂了 | celery/SSE 全停；`docker restart bjt-redis`；broker 队列在 db1、SSE Stream(`sse:stream:*`)在 db0，重启后均可续 |
| **NFS（node1）挂了** | **全集群文件读写失败**（workspace 在 node1）——上传/解析/读图都受影响。`systemctl status nfs-kernel-server`（node1）、`mountpoint /home/bjt/bjt_agent/workspace`（node2/3）。node1 是存储单点，短期接受，长期可加 NFS HA 或转对象存储 |
| LLM 429 限流 | Redis 分布式限流器自动排队；`docker exec bjt-redis redis-cli -n 1 --scan --pattern 'metrics:llm_429:*'` 看 429 计数；DeepSeek/百度 OCR 余额不足时充值 |
| 5xx 突增 | 先 `nginx -t`，再考虑回滚（见 §7）；查节点 `nodeN_backend_error.log` |
| celery `DuplicateNodenameWarning celery@bjt-check` | prod 老问题（prod celery 没设 `--hostname`，多个 worker 同名）；仅告警不影响。节点已用唯一名 `nodeN_review_i` |

## 7. 回滚到单机（秒级）

集群出问题、要回到切流量前的单机态：
```bash
# prod 上（bjt.precluster.bak 是切流量前的备份，prod 旧计算一直在跑）
cp /etc/nginx/sites-available/bjt.precluster.bak /etc/nginx/sites-available/bjt
nginx -t && systemctl reload nginx
```
reload 后流量全回 prod 本机 gunicorn:8001。节点服务不用动（可留着，也不影响）。

## 8. Phase 4：停 prod 计算角色（确认稳定后再做）

目标：prod 只留 nginx-LB + DB + Redis + PgBouncer + RAG + beat，停掉本机 gunicorn + review/parser celery（避免和节点抢资源、清晰职责）。

```bash
# 1) 确认无在途任务、队列空
docker exec bjt-redis redis-cli -n 1 LLEN review   # =0
docker exec bjt-redis redis-cli -n 1 LLEN parser   # =0
# 2) 确认 nginx 已全量切到节点（§7 的反方向，已在上次 cutover 完成）
# 3) prod 上停计算（保留 RAG + beat）
su - bjt -c 'cd /home/bjt/bjt_agent && \
  # 只停 backend + review + parser，手动留 beat + rag
  pkill -f "gunicorn backend.main:app"; \
  pkill -f "celery.*-Q review"; pkill -f "celery.*-Q parser"'
# 4) 确认 beat / RAG 仍在、节点继续接流量
./scripts/bjt-proc.sh status
for n in node1 node2 node3; do ssh bjt-$n curl -s http://127.0.0.1:8001/health; done
```
> 没有专用脚本只停"部分"——`bjt-proc.sh stop` 会全停（含 beat/RAG）。所以 Phase 4 用 `pkill` 精确停，或先 `stop` 再单独起 beat+RAG。**留 beat 在 prod 跑**（`celery -A celery_app beat`，全集群唯一）。

## 10. 已知坑（踩过的）

- **celery/rag 起不来** → `su - bjt -c '...start'` 非交互式时 conda/nvm 没激活。必须交互式 login shell 或手动 `source ~/miniconda3/etc/profile.d/conda.sh && conda activate py311`（节点的 `bjt-cluster.sh` 已自带激活，prod 的 `bjt-proc.sh` 同理——但调用方 shell 要先进 bjt）。
- **节点 `.env` 不能从 prod 照抄** → DB/Redis host 不同（prod `localhost`、节点 `10.1.3.241`）。用 `sales/scripts/cluster-deploy/16_node_env.py` 生成。
- **节点不通 GitHub/PyPI** → 代码/依赖从 prod rsync，别在节点上 `git pull`/`pip install`。
- **prod 部署的 `review_tasks.py`/`document_parser.py` 比 worktree 旧**（prod master 落后 sales 分支）→ 改这两文件时以 prod 实际版本为准，rsync 回节点；worktree 的新版本含 `utc_seconds_between` 等 prod 还没有的依赖。
- **TZ** → 节点已设 `Asia/Shanghai`；但 celery 进程启动在设 TZ 之前的话，mingle 会报 8h drift（仅显示）。重启 celery 即消。
- **前端 `/home/projects/.../images/*.jpeg` 404** → 切流量前就有的既有问题（nginx 从 `frontend/dist` 找这些图，但图在 workspace），与集群无关。
