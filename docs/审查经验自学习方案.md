# 标书审查 Agent 经验自学习方案

> 版本：v2.0 | 日期：2026-06-01 | 编写：技术团队
>
> 参考理论：《Agent 的常见范式》（ReAct / Plan-and-Execute / Reflection / Multi-Agent / Agentic Workflow）

---

## 一、背景与问题

当前标书审查系统每次审查任务**独立运行**，使用相同的静态规则文档和 Prompt，历史审查结果仅做去重合并，不会反馈到后续审查行为中。这带来三个核心问题：

1. **同类错误反复出现**：例如某类招标文件的盖章要求经常被漏检，严重程度判断不一致，但系统无法"记住"这类教训。
2. **领域知识无法积累**：已完成的数百次审查中蕴含着宝贵的领域经验（如"某行业技术偏离通常标 major 而非 critical"），但这些经验没有被系统化保存和复用。
3. **审查质量无法量化**：目前缺少客观的质量评估机制，无法回答"我们的审查准确率是多少"这个基本问题。

**核心诉求**：让 AI 审查系统具备"越用越准"的能力——从每次审查中提取经验，存储到记忆体中，在后续审查中参考历史经验，实现自我迭代优化。

## 二、Agent 范式分析与系统定位

> 本节基于 Agent 工程领域的主流范式框架，分析我们当前系统的技术定位和缺失环节。

### 2.1 五大 Agent 范式概览

业界 Agent 系统从简单到复杂，形成了五种核心范式，它们各自解决了不同阶段的核心痛点：

| 范式 | 一句话总结 | 解决的核心问题 |
|------|-----------|---------------|
| **ReAct** | 走一步，看一步，边想边做 | 让 Agent 能动起来——任务过程不确定，无法预知需要哪些步骤 |
| **Plan-and-Execute** | 先规划做什么，再一步一步做 | 长任务不迷路——防止 Agent 偏离目标、重复劳动 |
| **Reflection** | 做完之后，回头看看做得好不好 | 输出质量不稳定——Agent 只向前走，不回头看 |
| **Multi-Agent** | 一个人做不完的活，分给几个人一起做 | 单 Agent 能力有限——上下文爆炸、顾此失彼 |
| **Agentic Workflow** | 预定义好步骤，模型一步一步做 | 生产环境不可控——权限、合规、审计、成本、失败兜底 |

用一个工程类比来理解这些范式的关系：

- **ReAct** = 函数调用：处理单个具体操作，灵活但缺乏全局结构
- **Plan-and-Execute** = 模块划分：先把系统拆成功能模块，再逐一实现
- **Reflection** = 单元测试：对每个模块的输出进行质量检查
- **Multi-Agent** = 微服务架构：把复杂系统拆成独立服务，各司其职
- **Agentic Workflow** = CI/CD 流水线：定义整体执行流程和规则，确保稳定可控

### 2.2 我们当前系统的范式定位

对标上述框架，分析我们标书审查系统当前的范式实现情况：

| 范式 | 当前实现状态 | 对应组件 |
|------|-------------|----------|
| ✅ **ReAct** | **已实现** | `BidReviewAgent` 继承 Mini-Agent 的 ReAct 循环——思考→调用工具→观察结果→继续思考 |
| ✅ **Plan-and-Execute** | **已实现** | `MasterAgent` 扫描规则库 → 创建 TodoItem → 分发给子 Agent 执行 |
| ✅ **Multi-Agent**（指挥官-工人模式） | **已实现** | `MasterAgent`（指挥官）+ 多个 `BidReviewAgent`（工人），每个子 Agent 负责一个规则文档 |
| ⚠️ **Agentic Workflow** | **部分实现** | 规则文档定义了审查步骤，但缺少人工审核节点和失败兜底机制 |
| ❌ **Reflection** | **完全缺失** | **系统只向前走，不回头看**——审查结果产出后没有质量检查、没有自我修正、没有经验积累 |

**关键发现：** 我们的系统已经实现了 ReAct + Plan-and-Execute + Multi-Agent 三种范式的组合，但**唯独缺少 Reflection（反思）环节**。这正是经验自学习方案要补齐的拼图。

### 2.3 从范式选型四问看我们的需求

Agent 范式选型有一个经典四问框架，我们逐条分析：

**问题 1：步骤能否事先列全？**
> ✅ **能。** 我们的 16 个规则文档（A001-A009、B001、C001-C002、D001-D003、E001）预先定义了所有检查项，每个检查项的名称、规则描述、正例、反例都已明确。
>
> → **结论：适用 Plan-and-Execute + Agentic Workflow。** 我们已正确选择了这条路线。

**问题 2：错了代价多大？**
> ⚠️ **很高。** 审查遗漏可能导致废标（直接经济损失数十万至数百万），误判严重程度可能导致投标策略失误。标书审查是一个"错了代价极高"的场景。
>
> → **结论：必须引入 Reflection + 规则校验，最好加人工审核节点。** 这正是我们缺失的关键环节。

**问题 3：是否需要多专业角色？**
> ✅ **需要。** 合规性检查（A类）、资质审查（B类）、星级条目（C类）、评分项（D类）、签章检查（E类）属于不同专业领域。
>
> → **结论：Multi-Agent 已是正确选择。** 每个子 Agent 专注一个规则文档，正是"指挥官-工人"模式。

**问题 4：合规与审计要求？**
> ⚠️ **强。** 标书审查涉及招投标法规，审查过程需要可追溯、可审计、可解释。
>
> → **结论：以 Workflow 为主，Agent 能力收进边界内。** 经验注入不应改变核心规则逻辑，只能作为补充指导。

### 2.4 结论：补齐 Reflection 是当前最高优先级

基于以上分析，我们当前系统的**最优技术路径**是：

```
当前：Plan-and-Execute + ReAct + Multi-Agent（已实现）
  ↓
目标：Plan-and-Execute + ReAct + Multi-Agent + Reflection（本方案）
  ↓
远期：Plan-and-Execute + ReAct + Multi-Agent + Reflection + Agentic Workflow（生产增强）
```

**遵循"最小必要复杂度"原则**——我们的系统已经正确选择了较复杂的 Multi-Agent 架构，现在需要做的是在已有架构上补齐 Reflection 环节，而不是引入更多范式。

## 三、现有基础评估

经过对系统架构的全面梳理，我们已具备实现 Reflection（反思）环节的关键基础设施：

| 已有组件 | 当前状态 | 在 Reflection 中的角色 |
|----------|----------|----------------------|
| **QualityEvaluator（质量评估器）** | 已开发完成，但未接入审查管线 | **反思评估器**——已实现合规准确性(40%)、严重度适当性(30%)、完整性(30%)三维评分 |
| **RAGSearchTool（知识检索）** | 已对接外部 rag_memory_service | **经验检索器**——支持向量+文本混合检索，可复用于从记忆体中检索经验 |
| **rag_memory_service（RAG 服务）** | 已稳定运行 | **经验记忆体**——支持文档索引、语义搜索、多租户隔离，可直接存储经验文档 |
| **ReviewResult（审查结果数据）** | 每次审查自动产出 | **反思输入**——包含需求、应标内容、合规判定、严重度、说明、建议等完整数据 |
| **AgentStep（执行轨迹）** | 每次审查自动记录 | **行为诊断**——记录了 Agent 每步思考、工具调用、结果，可用于诊断行为模式 |

**结论：技术可行性高，Reflection 所需的"评估器"和"记忆体"均已就绪。**

## 四、三种技术方案

> 以下三种方案本质上都是 **Reflection 范式** 的不同实现策略，区别在于"谁来当裁判"和"如何用经验"。

### 方案 A：自动反思 + 经验检索（推荐优先实施）

> 范式定位：Reflection（LLM 自评） + RAG 经验回注

#### 工作原理

```
审查完成 → 质量评估器评分（反思）→ 经验提取器分析 → 存入 RAG 知识库（记忆体）
                                                      ↓
下次审查 → 系统Prompt构建 → 检索相关历史经验 → 注入审查指导（经验回注）
```

这正好对应 Reflection 范式的核心循环——**生成 → 反思 → 修正**，只不过修正不是在当次任务内完成，而是跨任务实现：

- **生成**：BidReviewAgent 执行审查，产出 ReviewResult
- **反思**：QualityEvaluator 对结果进行三维评分，ExperienceExtractor 提取结构化经验
- **修正**：下次审查时，将历史经验注入 Prompt，指导 Agent 避免同类错误

**具体流程：**

1. **事后反思**：每次审查任务完成后，调用已开发的 QualityEvaluator 对审查结果进行三维评分（合规准确性、严重度适当性、完整性）。
2. **经验提取**：LLM 分析评分结果和审查过程，自动提取结构化经验。例如：
   - "规则 A008 投标报价检查中，多地点报价一致性项经常被遗漏，需重点关注表格数据对比"
   - "规则 B001 资质审查中，过期证书应标 critical 而非 major"
3. **经验存储**：将提取的经验文档存入 rag_memory_service，按规则文档分类索引。
4. **经验注入**：下次审查同类规则时，自动检索相关经验，以"审查指导"形式注入到 Agent 的系统 Prompt 中。

#### 优势

- **自动化程度高**：无需人工干预，每次审查自动积累经验
- **复用现有组件**：QualityEvaluator 和 RAG 服务均已开发完成
- **非侵入式**：经验作为补充指令注入，不修改原始规则和 Prompt，符合 Agentic Workflow 的可控性要求
- **即时回滚**：可一键禁用经验注入功能，恢复原始行为

#### 不足

- **冷启动**：前几次审查无历史经验可用（可通过规则文档中的正例/反例播种缓解）
- **LLM 自评偏见**：AI 评估自身的准确性有限——正如 Agent 范式文章指出的，"如果反思不够准确，可能会反复修改同一个问题"
- **Context 窗口占用**：需控制经验注入量（建议每次不超过 3-5 条）

#### 预期效果

| 指标 | 当前基线 | 3个月后预期 |
|------|----------|------------|
| 合规准确性 | ~75-80% | ~82-87% |
| 严重度适当性 | ~65-70% | ~75-80% |
| 检查完整性 | ~70-75% | ~80-85% |

---

### 方案 B：用户反馈 + 模式提取（建议作为方案 A 的补充）

> 范式定位：Reflection（人类裁判） + Agentic Workflow（人工审核节点）

#### 工作原理

```
用户查看审查结果 → 标记"同意/不同意" + 修正建议 → 存入反馈库（人工审核节点）
                                                      ↓
定期分析反馈数据 → 提取共性问题 → 更新经验库
```

这对应了 Agent 范式文章中 **Agentic Workflow** 的关键理念——"在关键节点设置人工审核和兜底机制"。用户反馈就是在 Reflection 循环中引入人类裁判，解决 LLM 自评偏见问题。

**具体流程：**

1. **反馈收集**：在审查结果页面的每条发现旁增加"同意 ✓ / 不同意 ✗"按钮和可选的修正意见输入框。
2. **反馈存储**：将用户反馈（同意/不同意、修正的严重度、修正说明）存入数据库。
3. **模式分析**：当某类发现被多次标记为"不同意"时，系统自动提取共性模式。例如：
   - 5位用户中有4位将规则 A007 的严重度从 major 修正为 critical → 生成经验"A007 保证金不足应为 critical"
4. **经验校准**：用户反馈生成的经验获得更高的置信度权重，可覆盖自动生成的经验。

#### 优势

- **信号质量最高**：人类专家验证过的经验，准确性远超 AI 自评，有效防止负学习
- **可量化评估**：用户同意率可直接衡量系统质量（如"当前同意率 87%"）
- **符合 Workflow 理念**：在关键节点引入人工审核，提升生产环境可控性

#### 不足

- **依赖用户参与**：需要用户主动提供反馈，实际使用中参与度可能不高
- **反馈循环慢**：需要积累足够的反馈量才能提取模式，不适合快速迭代
- **需要前端改造**：审查结果页面需要新增反馈交互组件

---

### 方案 C：对抗式双 LLM 审查（资源消耗最大）

> 范式定位：Reflection（对抗式 Multi-Agent）——引入"质疑者"角色

#### 工作原理

```
主Agent产出审查结果 → 对抗审查LLM（专门"挑错"和"找漏"）
                            ↓
              产出三类信号：争议发现 / 遗漏发现 / 置信度校准
                            ↓
                   结构化为经验存入经验库
```

这对应了 Multi-Agent 中的**圆桌讨论模式**——所有 Agent 地位平等，通过互相辩论来提升决策质量。但正如 Agent 范式文章的警示：**"很多人一开始就想做一个无所不能的 Multi-Agent 系统，结果发现调试困难、成本高昂、稳定性差，最后还不如一个简单的 ReAct Agent 好用。"**

**具体流程：**

1. **主审查完成**：主 Agent 按正常流程产出审查结果。
2. **对抗审查**：启动第二个 LLM（可以使用相同模型），赋予"质疑者"角色：
   - **质疑已有发现**：检查每条"不符合"判定是否成立，识别误报
   - **寻找遗漏**：检查是否有应该发现但未发现的问题，识别漏检
   - **置信度评估**：判断哪些发现是"明确确定"的，哪些是"模棱两可"的
3. **经验生成**：将对抗审查中发现的误报模式和漏检模式结构化为经验。

#### 优势

- **全自动**：不依赖任何人工干预
- **双维度改善**：同时提高精确率（减少误报）和召回率（减少漏检）
- **可离线运行**：对抗审查可在后台异步执行，不影响前台审查速度

#### 不足

- **成本翻倍**：每次审查需要两倍 LLM API 调用量
- **复杂度高**：对抗审查者本身也可能出错——"如果反思不够准确，可能会反复修改同一个问题"
- **违反最小必要复杂度原则**：在方案 A 已实施的情况下，方案 C 增加的范式复杂度远大于其边际效益

## 六、推荐实施路径

### 推荐策略：A + B 分阶段混合实施

遵循**最小必要复杂度原则**——方案 A 补齐 Reflection 环节（LLM 自评），方案 B 引入人类裁判校准（Agentic Workflow 的人工审核节点）。两者互补：

- **A 解决"快速起效"**：自动反思，无需人工干预，2-3 周见效
- **B 解决"长期可靠"**：人类校准，防止负学习，3-5 周见效
- **不建议 C**：成本翻倍、复杂度高，违反最小复杂度原则

```
  ├─ 建立三维评分基线     ├─ 经验存入 RAG 服务    ├─ 开发反馈 API         ├─ 经验衰减机制
  ├─ 前端展示质量趋势     ├─ 审查前检索经验       ├─ 前端反馈交互组件     ├─ 经验去重合并
  └─ 不改变审查行为       └─ 注入系统Prompt       └─ 反馈校准经验         └─ 质量仪表盘
```

#### Phase 1：建立度量基线（第 1 周）

**目标：** 不改变任何审查行为，仅为每次审查自动打分，建立量化基线。

**工作内容：**
- 将已开发的 QualityEvaluator 接入审查管线
- 数据库新增质量评分字段
- API 新增质量指标查询端点
- 前端展示审查质量趋势图

**风险：** 极低（仅增加评估，不修改审查逻辑）
**交付物：** 每次审查自动产出质量评分报告，可回答"审查准确率是多少"

---

#### Phase 2：自动反思 + 经验积累（第 2-4 周）

**目标：** 实现方案 A，补齐 Reflection 环节，让系统从每次审查中自动学习。

**工作内容：**
- 开发 ExperienceExtractor（经验提取器）：分析审查结果 + 质量评分，提取结构化经验
- 开发 ExperienceRetriever（经验检索器）：审查前按规则文档检索相关经验
- 修改系统 Prompt 模板，在规则文档之后注入历史经验指导
- 经验存入 rag_memory_service，支持按规则文档精确检索

**风险控制：**
- 经验置信度门槛：至少 3 次审查贡献且置信度 ≥ 0.7 的经验才进入活跃使用
- 经验注入量控制：每次不超过 3-5 条，每条不超过 150 字
- 一键回滚：通过特性开关控制，可随时禁用经验注入

**交付物：** 系统具备"越审越准"能力，同类项目审查质量逐步提升

---

#### Phase 3：用户反馈闭环（第 5-7 周）

**目标：** 实现方案 B，在 Reflection 循环中引入人类裁判（Agentic Workflow 的人工审核节点）。

**工作内容：**
- 数据库新增 `finding_feedback` 表
- 开发反馈提交 API
- 前端审查结果页新增"同意/不同意"反馈按钮
- 开发反馈分析服务：从用户修正中提取模式，更新经验库

**交付物：** 用户可直接纠正 AI 审查结果，纠正信号自动反馈到未来审查中

---

#### Phase 4：验证与优化（第 8-10 周）

**目标：** 量化验证效果，持续优化。

**工作内容：**
- 构建 A/B 对比评估：同一项目分别用"有经验注入"和"无经验注入"审查，对比质量评分
- 经验衰减机制：30 天未被验证的经验自动降权
- 经验去重合并：相似经验自动合并
- 质量仪表盘：可视化展示质量趋势、用户同意率、经验使用情况

**交付物：** 完整的效果验证报告和持续优化机制

## 八、风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |

## 九、结论与建议

1. **该方案技术可行**：核心组件（QualityEvaluator、RAG 服务、结构化审查数据）已就绪，不需要大规模基础建设。

2. **理论框架支撑**：对标 Agent 范式框架，我们的系统已实现 Plan-and-Execute + ReAct + Multi-Agent，**经验自学习本质上是补齐缺失的 Reflection 环节**——这是从"能审查"到"审查好"的关键跃迁。

3. **遵循最小必要复杂度原则**：方案 A（自动反思）以最低的范式复杂度补齐 Reflection，方案 B（用户反馈）以 Agentic Workflow 的人工审核节点保证长期可靠性。不建议方案 C，因其违反最小复杂度原则。

4. **建议优先启动 Phase 1**：建立度量基线是零风险快赢，1 周可交付，且为后续所有学习机制奠定基础。在此之后可根据 Phase 1 的质量基线数据，决定 Phase 2 的优先级和投入力度。

5. **实施后可实现质变**：系统从"每次从零开始"升级为"越审越准"，**补齐 Reflection 后，我们的系统将实现五种 Agent 范式的完整组合**——这在同类产品中是领先的技术架构。
