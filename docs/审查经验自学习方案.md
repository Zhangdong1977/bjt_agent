## 一、背景与问题

### 1.1 问题陈述

标书审查系统每次任务独立运行，使用相同的静态规则文档和 Prompt，历史审查结果仅做去重合并，不会反馈到后续审查行为中。三大痛点：

1. **同类错误反复出现**：盖章要求漏检、严重度判断不一致，系统无法"记住"教训
2. **领域知识无法积累**：数百次审查中蕴含的领域经验未被系统化保存
3. **审查质量无法量化**：缺少客观评估机制，无法回答"审查准确率是多少"

### 1.2 核心思路

补齐 **Reflection 范式**：审查完成后自动反思 → 提炼可复用经验 → 检索注入到下次审查的 Prompt 中，形成闭环。

### 1.3 需要解决的六个工程问题

| 问题 | 说明 |
|------|------|
| **经验条目的数据模型** | 需要统一的字段定义，支持跨条目复用与精确检索 |
| **经验的产生机制** | LLM 输出需要结构化约束，防止格式漂移和信息丢失 |
| **经验的聚类与去重** | 防止经验库膨胀为长尾重复条目 |
| **负学习防护** | 错误经验混入后需要能被压制，防止污染后续审查 |
| **经验的过期与淘汰** | 避免历史包袱累积，与最新规则文档脱节 |
| **经验与规则的耦合方式** | 需要明确的隔离边界，防止跨规则污染 |

### 1.4 设计约束

- **复用现有基础设施**：QualityEvaluator、rag_memory_service、PostgreSQL、Redis 全部沿用
- **不破坏现有审查行为**：经验注入作为非侵入式补充，可一键关闭
- **不直接 import EverOS**：EverOS 作为 `third_party` 子模块，仅参考其数据模型与 Prompt 设计，代码自实现

## 二、EverOS 六大核心机制原理

> 本章节是方案的理论基础。全部内容来自对 [third_party/EverOS/methods/EverCore/](../third_party/EverOS/methods/EverCore/) 源码的解读，引用具体文件与行号。

### 2.1 机制一：`AgentCase` —— 微观经验的最小单元

#### 原理

`AgentCase`（智能体案例）是 EverOS 中"一次问题求解过程"的压缩表示。它与原始对话/工具调用轨迹不同——**只保留跨任务可复用的那部分信息**。

定义在 [memory_models.py:344](../third_party/EverOS/methods/EverCore/src/api_specs/memory_models.py#L344)：

```python
@dataclass
class AgentCaseModel:
    id: str
    timestamp: datetime
    # 核心字段
    task_intent: str          # 任务意图：自包含的一句话描述"做了什么"
    approach: str             # 方法：分步骤的"尝试了什么 + 结果是什么"
    quality_score: float      # 质量分：0-1，LLM + 评估器综合打分
    # 父子链接（指向触发它的 MemCell）
    parent_type: str
    parent_id: str
    # 多租户隔离
    user_id, group_id, session_id: str
```

> **注意**：EverOS 的 `AgentCaseModel` 不含 `key_insight` 字段。`key_insight` 出现在 `AGENT_CASE_COMPRESS_PROMPT` 的输出 JSON 中（[agent_prompts.py:150](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L150)），是压缩过程产出的中间字段。本地化方案将其持久化到 `experience_cases` 表。

#### 关键设计思想

1. **压缩比 ~10%**：原始 AgentStep 轨迹（含所有工具调用、工具返回、思考）可能上万 token，AgentCase 压缩到几百 token。压缩由 `AGENT_TOOL_PRE_COMPRESS_PROMPT` + `AGENT_CASE_COMPRESS_PROMPT` 两阶段完成。
2. **可被聚类**：`task_intent` 是后续聚类的语义锚点（嵌入向量的输入文本）。
3. **可被评分**：`quality_score` 决定该 Case 是进入"成功提取 Prompt"还是"失败提取 Prompt"（详见 2.6）。
4. **可溯源**：`parent_id` 指回原始 MemCell，需要审计时可还原完整轨迹。

#### 为什么不是直接存原始 ReviewResult

原始 ReviewResult 包含大量本次任务专用信息（招标文件名、应标表格内容、具体页码）。这些信息**对同类任务无复用价值**，反而会污染后续 Skill 提取的 LLM 上下文。AgentCase 的压缩过程起到了**信息过滤**的作用：保留"可复用的方法论"，丢弃"一次性的细节"。

### 2.2 机制二：`AgentSkill` —— 宏观经验的可执行 SOP

#### 原理

`AgentSkill`（智能体技能）是从一类相似的 `AgentCase` 中提炼出的可执行 SOP（Standard Operating Procedure）。**Skill 不是案例的简单摘要，而是对案例中可复用步骤的泛化**。

定义在 [memory_models.py:373](../third_party/EverOS/methods/EverCore/src/api_specs/memory_models.py#L373)：

```python
@dataclass
class AgentSkillModel:
    id: str
    cluster_id: str           # 所属 Skill 簇（由聚类机制决定）
    name: str                 # 技能名称（≤10 词）
    description: str          # 一句话抽象描述 + Keywords
    content: str              # Markdown SOP 主体（## Steps + ## Pitfalls）
    confidence: float         # 置信度（详见 2.4）
    maturity_score: float     # 成熟度（详见 2.5）
    agent_case_ids: List[str] # 溯源到所有贡献过该 Skill 的 Case
    user_id, group_id: str
```

> **注意**：EverOS 的 `AgentSkillModel` 默认 `maturity_score = 0.6`。本地化方案将默认值改为 `0.0`，确保新 Skill 在首次评分前不会误入 Prompt。

#### 双模形态（核心创新）

EverOS 强制把 Skill 分成两种**模板形态**，从结构上就能一眼区分"已验证"与"假设性"：

| 形态 | 触发条件 | 模板结构 | 语义 |
|------|---------|---------|------|
| **已验证 SOP** | 来源 Case 全部 `quality_score ≥ 0.5` | `## Steps`（含 How / Decision / e.g. / Check） + 可选 `## Pitfalls` | 这些步骤已被实战验证，可直接复用 |
| **假设性 SOP** | 来源 Case 全部 `quality_score < 0.5` | `## Potential Steps`（带"未验证"声明）+ 强制 `## Pitfalls` | 这些步骤只是失败案例中的局部进展，复用需谨慎 |

#### 为什么这样设计

- **结构化防止 LLM 撒谎**：失败的步骤如果被无差别地放进 `## Steps`，后续 Agent 会当作"已验证"复用，导致错误传播。强制写进 `## Potential Steps` + 警示性声明，让下游 Prompt 注入时也能传递这层语义。
- **支持假设晋升**：当一条 `## Potential Steps` Skill 后续被成功案例验证，会自动重写为 `## Steps` 并重评分（详见 2.5）。这是 EverOS 实现"从失败中学习并渐进验证"的核心机制。

### 2.3 机制三：在线增量聚类 —— Skill 簇的形成

#### 原理

EverOS 不预先定义 Skill 类别，而是**让 Skill 簇随着新 Case 的到来动态形成**。新 Case 进入时，需要决定它属于哪个现有簇（合并到现有 Skill）还是新建一个簇（产生新 Skill）。

实现位置：[manager.py:404](../third_party/EverOS/methods/EverCore/src/memory_layer/cluster_manager/manager.py#L404) 的 `_cluster_memcell_llm` 方法。

#### 两阶段判定

1. **嵌入召回（快速过滤）**
   - 计算新 Case 的 `task_intent` 嵌入向量
   - 与现有所有 Case 簇的质心（centroid）做余弦相似度
   - 取 top-K 候选簇（默认 K=5）
   - 如果 top-1 相似度 ≥ `llm_skip_threshold`（默认 0.85，见 [config.py:32](../third_party/EverOS/methods/EverCore/src/memory_layer/cluster_manager/config.py#L32)），直接接受，跳过 LLM

2. **LLM 域校验（防止跨域误合并）**
   - 把候选簇的代表 Case 与新 Case 一起喂给 LLM
   - LLM 从两个维度判断：
     - **主题域（subject domain）**：是否属于同一业务领域
     - **问题模式（problem-solving pattern）**：是否采用相似的解决思路
   - **硬约束：禁止跨域合并**——即使嵌入相似度高，不同领域也不能并入同一簇

#### 关键设计思想

- **在线增量，不批量重聚类**：每次新 Case 触发一次聚类决策，不需要周期性 batch job。质心是滑动平均，新 Case 进入后增量更新。
- **LLM 兜底，嵌入兜底 LLM**：嵌入快速过滤解决 90% 简单情况，LLM 解决 10% 边缘情况（嵌入误判）。两者互补。
- **跨域硬约束**：纯嵌入容易把"A001 资质审查"和"B001 财务审计"并入一簇（都是"检查证书"模式），但语义上完全不同。LLM 域校验阻止这种情况。

### 2.4 机制四：置信度锚定 —— 经验的可信度演进

#### 原理

EverOS 不用"信任 / 不信任"二元的标签，而是给每个 Skill 一个 [0, 1] 区间的**置信度**（confidence），并通过**事件驱动**的方式让置信度随新证据持续演化。

实现位置：`AGENT_SKILL_SUCCESS_EXTRACT_PROMPT` 第 237-243 行的 "Confidence Anchoring Rules" 段——这是**直接写在 Prompt 里、让 LLM 来执行**的规则（而非硬编码逻辑）。

#### 锚定规则（核心）

| 事件 | 置信度变化 | 含义 |
|------|-----------|------|
| **新建 Skill（add）** | confidence = **0.5** | 中性起点——既不预设可信也不预设可疑 |
| **假设晋升（hypothesis → verified）** | confidence = **0.6** | 从 `## Potential Steps` 升级为 `## Steps`，获得小幅背书 |
| **新增决策分支（update with new branch）** | confidence += **0.1**（cap 0.95） | Skill 被新案例扩展，可信度上升 |
| **纯确认（confirming update）** | confidence += **0.05**（cap 0.95） | 新案例与现有 Skill 完全吻合，弱证据增强 |
| **矛盾案例（contradicting case）** | success: −0.2 / failure: −0.15~−0.25 | 新案例与 Skill 直接冲突，强证据削弱 |
| **退休线（retire）** | confidence < **0.1** | 从检索引擎移除，但数据库保留（可回滚） |

> **注意**：矛盾案例的置信度惩罚在 EverOS 中是**按 Prompt 区分的**——`SKILL_SUCCESS_EXTRACT_PROMPT` 使用固定值 −0.2，`SKILL_FAILURE_EXTRACT_PROMPT` 使用范围 −0.15~−0.25（由 LLM 根据矛盾程度判定）。

#### 关键设计思想

1. **初始值 0.5，不是 0**：新经验不是"有罪推定"也不是"无罪推定"，给一个中性起点，让证据自己说话。
2. **不对称更新**：confirm 仅 +0.05，contradict 却 −0.2。**4 个 confirming 才抵得上 1 个 contradict**。这反映"证伪比证实容易"的科学哲学（波普尔）——错误经验应该容易被纠正。
3. **写在 Prompt 里，不是硬编码**：EverOS 的设计哲学是"让 LLM 执行软规则，让代码执行硬约束"。LLM 看到 "Confidence Anchoring Rules" 段会按规则输出 confidence 值，代码只负责把这个值持久化。优点：规则可以随 Prompt 升级而进化，不需要改代码。
4. **cap 上限 0.95**：永远不达到 1.0，保留"任何 Skill 都可能被反驳"的可能性。

#### 标书审查场景的意义

`A007 保证金不足应为 critical`——这条经验第一次出现时 confidence=0.5。如果用户反馈连续 3 次确认，confidence 升到 0.65。但如果 1 次用户反馈反对，confidence 跌到 0.45。**经验的可信度随实际使用动态演进，而不是一次写入永久不变**。

### 2.5 机制五：成熟度门控 —— 多维度质量评分

#### 原理

如果说 `confidence` 衡量"这条经验多可信"，`maturity_score` 衡量的是"这条经验多好用"——一条 Skill 可能绝对可信（来自 100 个成功案例），但写得糟糕（缺步骤、缺示例、组织混乱），仍然不应该被注入到下次审查的 Prompt 中。

实现位置：`AGENT_SKILL_MATURITY_SCORE_PROMPT`（[agent_prompts.py:370](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L370)），由 `AgentSkillExtractor` 在 Skill 内容发生实质变化时调用。

#### 四轴评分（每轴 1-5 分，归一化到 0-1）

| 维度 | 含义 | 低分示例 | 高分示例 |
|------|------|---------|---------|
| **Completeness（完整性）** | SOP 是否端到端覆盖完整流程 | "检查资质文件"——缺具体步骤 | 完整的发现→验证→定级→建议流程 |
| **Executability（可执行性）** | Agent 能否不猜地按 SOP 操作 | "分析问题"——无具体动作 | "用 DocSearchTool 查询证书过期日 → 与招标要求对比" |
| **Evidence（证据）** | 是否有充分的案例支撑 | 无任何案例示例 | 多步骤跨场景的真实示例 |
| **Clarity（清晰度）** | 结构是否清晰、Markdown 是否规范 | 一坨散文 | 干净的标题、编号、子项、可扫描 |

#### 双模差异化评分

同一个 Prompt 对两种形态采用不同基线：
- **`## Steps`（已验证）**：按完整 SOP 标准评——完整性要求高
- **`## Potential Steps`（假设性）**：按"失败教训"标准评——完整性本身受限，更看 Pitfalls 是否充实

#### 门控阈值

- **maturity_score ≥ 0.6** → 活跃，允许注入到下游 Prompt
- **maturity_score < 0.6** → 沉默，存在但不可见，等待更多案例提升

#### 重评分时机（成本控制）

不是每次 Skill 更新都触发重评分，避免 LLM 成本爆炸：
- 内容变化比 < 20%：跳过
- 内容变化比 ≥ 40%：强制重评分
- 假设晋升（`## Potential Steps` → `## Steps`）：强制重评分
- 中间区（20-40%）：仅当 Skill 不成熟或置信度下降时重评分

#### 标书审查场景的意义

`"保证金检查要看账户类型"`——这条 Skill 即使被 50 个案例确认（confidence=0.95），如果内容只写了"看账户类型"四个字，没有具体步骤、没有示例、没有决策分支，maturity_score 仍然 < 0.6，不会注入下次审查。**这避免了"可信但无用"的经验污染 Prompt**。

### 2.6 机制六：成功/失败双模提取 + Skill 退休

#### 原理

最后的"组合拳"——前面五个机制协同工作形成完整的 Skill 生命周期：

```
                  新 Case 进入
                       │
            ┌──────────┴──────────┐
            │ quality_score ≥ 0.5 │
            ▼                     ▼
        成功 Case              失败 Case
            │                     │
            ▼                     ▼
  SKILL_SUCCESS_EXTRACT    SKILL_FAILURE_EXTRACT
    产出/更新                产出/更新
   ## Steps SOP           ## Potential Steps
       +                      +
   (Pitfalls 可选)         ## Pitfalls (强制)
            │                     │
            └──────────┬──────────┘
                       ▼
            maturity_score 重评分
                       │
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
   ≥ 0.6 活跃      < 0.6 沉默       confidence < 0.1
   (注入 Prompt)   (等待证据)       (退休)
```

#### 双模提取的产出差异

| 特征 | SKILL_SUCCESS_EXTRACT | SKILL_FAILURE_EXTRACT |
|------|----------------------|----------------------|
| 输入 | 成功 Case（quality ≥ 0.5） | 失败 Case（quality < 0.5） |
| 产出结构 | `## Steps`（带 How/Decision/e.g./Check） | `## Potential Steps`（带"未验证"声明）+ `## Pitfalls`（强制） |
| 置信度锚定 | confirm +0.05 / branch +0.1 / contradict −0.2 | confirm +0.05 / 新增 pitfall 不变 / contradict −0.15~−0.25 |
| 关键约束 | Steps 最多 5 步，每步 ≤2 个示例 | Pitfalls 至少 1 条，必须可追溯到具体失败 |
| 注入语义 | "这是已验证的 SOP，请遵循" | "这是失败案例的局部进展，参考但谨慎" |

#### 假设晋升（Hypothesis Promotion）

最精妙的设计：**一条失败 Skill 可以通过新成功案例"晋升"为已验证 Skill**。

由 `SKILL_SUCCESS_EXTRACT_PROMPT` 的 Hypothesis promotion rule 实现（[agent_prompts.py:232](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L232)）：

> If the existing skill contains `## Potential Steps`, treat this update as a **promotion** — rewrite as `## Steps` using the new case as primary source. confidence = `0.6`.

这意味着：
1. 失败案例先建立"假设性 SOP"（带 Pitfalls）
2. 后续成功案例如果落在同一 Skill 簇，LLM 检测到 `## Potential Steps` 标记
3. 自动改写为 `## Steps`，confidence 升到 0.6，触发成熟度重评分

**这正是"从失败中学习并渐进验证"的工程实现**。

#### Skill 退休（不是删除）

当 confidence < 0.1（默认值，可配置）：
- 从 rag_memory_service（检索引擎）**移除**——后续不再被检索、不再被注入
- PostgreSQL `experience_skills` 表**保留**——`retired_at` 字段标记退休时间
- 需要 audit 时仍可查询；如果发现是误判，可以"复活"（清空 `retired_at` 并重新索引）

实现位置：`AgentSkillExtractor` 的 `retire_confidence` 参数（默认 0.1）。

#### 标书审查场景的意义

假设第一次审查 A008 投标报价时，Agent 漏检了多地点报价一致性——这次审查 quality_score=0.3，进入失败模式，产出 `## Potential Steps`："检查多地点报价一致性，但尚未验证具体方法" + `## Pitfalls`："仅对比了主报价表，忽略了分支机构报价附表"。

后来另一次审查 A008 成功捕获了这个问题——quality_score=0.85，触发 Skill 更新。LLM 检测到现有 Skill 是 `## Potential Steps`，自动晋升为 `## Steps`："用 DocSearchTool 检索'分支机构'关键字 → 对比各分支机构报价表 → 与主表汇总数据一致性校验"，并继承之前的 Pitfalls。

下一次同类项目审查 A008 时，这条已晋升的 Skill 会被检索并注入到 Prompt，**Agent 从此不再漏检多地点报价一致性**。

## 三、本地化映射：从通用 Agent 到标书审查

### 3.1 概念映射表

| EverOS 原语 | 标书审查本地化含义 | 落地具体值 |
|------------|------------------|-----------|
| `MemCell`（记忆边界） | 一次 BidReviewAgent 完整审查（一个规则文档）的边界 | 由 `MasterAgent._run_single_sub_agent` 触发 |
| `AgentCase` | 一次单规则文档审查的压缩经验 | `task_intent` = "审查 A001 投标文件填写完整性"，`approach` = AgentStep 轨迹压缩 |
| `AgentSkill` | 一类规则文档的审查 SOP | 一个 Cluster 下的所有 Case 共同形成的"如何审查 A001"的可执行手册 |
| `Cluster` | 规则文档内的子类型聚类 | 例如 A001 下可能聚为"表格完整性"和"签字盖章完整性"两个子簇 |
| `quality_score` | 审查质量评分 | QualityEvaluator 三维加权得分 / 100，融合 LLM 主观评分 |
| `confidence` | 经验可信度 | 用户反馈 confirm/contradict 驱动 |
| `maturity_score` | SOP 可用度 | 完整性/可执行性/证据/清晰度四轴 |
| `group_id` | **隔离边界** | `rule_doc_stem`（如 `A001 检查投标文件填写完整性`） |
| `session_id` | 审查任务标识 | `ReviewTask.id` |
| `user_id` | 审查发起人 | `Project.user_id`（仅用于视图过滤，不参与检索过滤） |

### 3.2 多租户策略

**全局共享 + 用户视图**：
- 写入：每次 Case 必带 `group_id=rule_doc_stem`，跨项目跨用户写入同一 group
- 检索：默认按 `group_id` 精确过滤，**所有用户共享**同一规则文档的经验
- `user_id` 用途：仅用于前端"我贡献的经验"视图、审计回溯、贡献度统计

**理由**：标书审查是规则强相关而非用户强相关的领域。A001 的审查经验对所有审查 A001 的用户都有价值。隔离到用户粒度会显著拖慢经验积累速度。

### 3.3 QualityEvaluator 的角色升级

[quality_evaluation.py:177](../backend/agent/quality_evaluation.py#L177) 的 `QualityEvaluator` 当前是孤立的——本方案把它接入管线，作为 `quality_score` 的**主要来源**（而非 EverOS 默认的纯 LLM 主观评分）：

```
quality_score = 0.6 × (QualityEvaluator.overall_quality_score / 100)
              + 0.4 × (AGENT_CASE_COMPRESS_PROMPT 产出的 LLM 主观评分)
```

**权重理由**：
- QualityEvaluator 基于三维客观评分（合规准确性 0.40 + 严重度适当性 0.30 + 完整性 0.30），更稳定，权重高（0.6）
- LLM 主观评分来自 AgentStep 轨迹分析，能捕捉"过程质量"（如 Agent 是否反复重试、是否合理使用工具），权重低（0.4）
- 两源融合比单一源更鲁棒——单源容易被对抗性案例欺骗

权重作为可调参项，Phase 4 A/B 测试时可根据实际效果调整。

## 五、Prompt 设计

> 本章节列出需要实现的 7 条核心 Prompt，每条给出：
> - **EverOS 出处**：原 Prompt 文件 + 行号
> - **核心职责**：Prompt 解决什么问题
> - **标书审查适配点**：相对原 Prompt 的修改方向

### 5.1 `CASE_FILTER_PROMPT`（Case 过滤器）

- **出处**：[AGENT_CASE_FILTER_PROMPT](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L44)
- **职责**：判断本次审查是否值得提取为 Case。过滤掉：纯 chitchat、单轮 Q&A、过于琐碎无方法论价值的审查
- **适配点**：
  - 标书审查几乎每次都有方法论价值，过滤门槛应低于通用 Agent
  - 仅过滤"任务异常终止 + 0 finding + 0 step"的纯失败情况
  - 输入：AgentStep 摘要 + ReviewResult finding 数 + 任务状态
  - 输出：`{should_extract: bool, reason: str}`

### 5.2 `CASE_COMPRESS_PROMPT`（Case 压缩器）

- **出处**：[AGENT_TOOL_PRE_COMPRESS_PROMPT](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L3) + [AGENT_CASE_COMPRESS_PROMPT](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L80)
- **职责**：两阶段把 AgentStep 轨迹压缩为 `(task_intent, approach, key_insight, quality_score_llm)`
  - 阶段 1：工具调用预压缩（保留语义，丢弃原始返回的大段文本）
  - 阶段 2：整体压缩为四元组
- **适配点**：
  - 标书审查的工具集已知（DocSearchTool / RAGSearchTool / ComparatorTool / 文档结构工具），可在 Prompt 中显式列出
  - `task_intent` 模板："审查 {rule_doc_name}，发现 {finding_count} 项{severity_summary}"
  - `approach` 必须体现"使用什么工具查了什么关键字→得到了什么→判定依据"
  - `quality_score_llm` 评分细则需对齐 QualityEvaluator 三维（合规准确性 / 严重度适当性 / 完整性）

### 5.3 `CLUSTER_LLM_ASSIGN_PROMPT`（聚类 LLM 分配器）

- **出处**：[AGENT_CLUSTER_LLM_ASSIGN_PROMPT](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L421)
- **职责**：嵌入召回 top-K 候选簇后，由 LLM 判断"领域 + 问题模式"是否同源，禁跨域合并
- **适配点**：
  - 域定义明确：rule_doc_stem 已是天然域边界，Cluster 在 rule_doc 内部进一步按子模式聚类
  - 例如 A001 内可能聚为"表格完整性子簇"、"签章完整性子簇"
  - 输入：新 Case 的 task_intent + 候选簇的代表 Case（最多 3 个/簇）
  - 输出：`{assigned_cluster_id: str | "new", reason: str}`
  - 硬约束：禁止跨 group_id 合并（这是规则文档级别的硬隔离）

### 5.4 `SKILL_SUCCESS_EXTRACT_PROMPT`（成功 Skill 提取器）

- **出处**：[AGENT_SKILL_SUCCESS_EXTRACT_PROMPT](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L155)
- **职责**：从 quality ≥ 0.5 的 Case 中提取或更新 `## Steps` 形态的 Skill
- **适配点**：
  - Steps 上限改为 6 步（标书审查比通用 Agent 步骤略多）
  - 每步强制要求"工具名 + 关键字 + 判定依据"三元组
  - Decision 分支用标书术语（"若为单包件标段" vs "若为多包件标段"）
  - 置信度锚定规则原样保留（add=0.5 / branch +0.1 / confirm +0.05 / contradict −0.2）

### 5.5 `SKILL_FAILURE_EXTRACT_PROMPT`（失败 Skill 提取器）

- **出处**：[AGENT_SKILL_FAILURE_EXTRACT_PROMPT](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L260)
- **职责**：从 quality < 0.5 的 Case 中提取 `## Potential Steps` + `## Pitfalls`
- **适配点**：
  - Pitfalls 必须可追溯到具体的 ReviewResult 或 AgentStep
  - 标书审查特有 Pitfall 模板：漏检类型（如"忽略分支机构报价"）、误判类型（如"将 minor 误判为 critical"）、误用工具类型（如"ComparatorTool 输入了不对齐的两段文本"）
  - 矛盾案例置信度惩罚使用范围值 −0.15~−0.25（与 EverOS 原设计一致）

### 5.6 `SKILL_MATURITY_SCORE_PROMPT`（成熟度评分器）

- **出处**：[AGENT_SKILL_MATURITY_SCORE_PROMPT](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L370)
- **职责**：四轴评分（completeness / executability / evidence / clarity），归一化到 [0, 1]
- **适配点**：
  - Executability 维度的"具体动作"必须是 bjt_agent 工具集中存在的工具调用
  - Evidence 维度的"案例示例"必须来自真实 ReviewResult 引用
  - 双模差异化评分保留——`## Steps` 按完整 SOP 评，`## Potential Steps` 按失败教训评

### 5.7 `SKILL_RELEVANCE_VERIFY_PROMPT`（注入相关性校验）

- **出处**：[AGENT_SKILL_RELEVANCE_VERIFY_PROMPT](../third_party/EverOS/methods/EverCore/src/memory_layer/prompts/en/agent_prompts.py#L346)
- **职责**：审查前从经验库召回 top-K Skill 后，由 LLM 校验每条 Skill 与本次审查的相关性，过滤误召回
- **适配点**：
  - 输入：本次审查的招标文件元信息（行业、标段类型、包件数）+ 召回 Skill 列表
  - 输出：每条 Skill 的 relevance_score (0-1) + reason
  - 阈值：relevance_score ≥ 0.6 才注入 Prompt
  - 与 maturity_score 联合排序：`final_score = maturity × confidence × relevance`

## 六、置信度与成熟度机制

> 本章节给出置信度与成熟度机制的具体配置参数与运行规则。

### 6.1 置信度锚定规则

| 事件 | confidence 变化 | 触发条件 |
|------|----------------|---------|
| 新建 Skill | = 0.5 | SKILL_*_EXTRACT 输出 `add` |
| 假设晋升 | = 0.6 | SKILL_SUCCESS_EXTRACT 检测到 `## Potential Steps` → `## Steps` |
| 新增决策分支 | += 0.1（cap 0.95） | SKILL_SUCCESS_EXTRACT 输出 `update` + 新增 Decision 分支 |
| 纯确认 | += 0.05（cap 0.95） | SKILL_SUCCESS_EXTRACT 输出 `update` + 仅确认 |
| 矛盾案例（成功 Prompt） | −= 0.2 | SKILL_SUCCESS_EXTRACT 输出 `update` + 明确反驳 |
| 矛盾案例（失败 Prompt） | −= 0.15~0.25 | SKILL_FAILURE_EXTRACT 输出 `update` + 明确反驳（LLM 判定程度） |
| 失败案例追加 Pitfall | 不变 | SKILL_FAILURE_EXTRACT 输出 `update` + 仅新增 Pitfall |
| 失败案例扩展 Potential Steps | += 0.05 | SKILL_FAILURE_EXTRACT 输出 `update` + 新增 Potential Step |
| 用户 confirm | += 0.05 | experience_feedback `feedback_type='confirm'` |
| 用户 contradict | −= 0.2 | experience_feedback `feedback_type='contradict'` |
| 用户 refine（修正） | −= 0.1 + 触发 Skill 内容重写 | experience_feedback `feedback_type='refine'` |
| **退休** | （从检索引擎移除） | confidence < 0.1 |

### 6.2 成熟度门控规则

- **阈值**：`maturity_score ≥ 0.6` 才允许注入到下次审查的 Prompt
- **重评分时机**：
  - 内容变化比 < 20% → 跳过
  - 内容变化比 ≥ 40% → 强制重评分
  - 假设晋升（`## Potential Steps` → `## Steps`）→ 强制重评分
  - 内容变化比 20-40% → 仅当 maturity < 0.6 或 confidence 下降时重评分
- **初始值**：新建 Skill 时 maturity_score = 0.0，等首次评分后写入。**0.0 → 不会注入**——避免新 Skill 在评分前误入 Prompt。

### 6.3 Skill 退休

- **触发**：confidence < 0.1（默认 `retire_confidence=0.1`）
- **动作**：
  1. rag_memory_service 中删除对应文档（`rag_doc_id` 字段保留作为审计痕迹）
  2. PostgreSQL `experience_skills.retired_at = NOW()`
  3. 后续检索 SQL 强制过滤 `retired_at IS NULL`
- **复活**：管理员可手动清空 `retired_at` + 重新索引到 rag_memory_service
- **不可恢复的删除**：仅当 Skill 内容违反法规或包含敏感信息时，由管理员手动硬删（PostgreSQL DELETE）

### 6.4 假设晋升（Hypothesis Promotion）

由 `SKILL_SUCCESS_EXTRACT_PROMPT` 的 Hypothesis promotion rule 自动执行：

1. LLM 检测到现有 Skill 的 `content` 包含 `## Potential Steps`
2. LLM 自动改写为 `## Steps`，以新成功案例为 primary source
3. confidence 重置为 0.6
4. 触发强制 maturity 重评分
5. `last_promoted_at` 字段记录晋升时间
6. 写入审计日志（Skill 历史版本归档）

## 七、检索与注入：经验如何进入下一次审查

### 7.1 检索路径（审查前）

```
BidReviewAgent.run_review() 启动
  │
  ├─ 解析 rule_doc_path → rule_doc_stem (group_id)
  │
  ├─ 调用 ExperienceRetriever.retrieve(group_id, query)
  │     │
  │     ├─ SQL 查询 experience_skills
  │     │   WHERE group_id = ?
  │     │     AND retired_at IS NULL
  │     │     AND maturity_score >= 0.6
  │     │     AND confidence >= 0.1
  │     │   ORDER BY maturity_score * confidence DESC
  │     │   LIMIT 10
  │     │
  │     ├─ 向量召回（rag_memory_service hybrid search）
  │     │   query = task_intent（或招标文件摘要）
  │     │   filters = {category: experience_skill, group_id}
  │     │   top_k = 10
  │     │
  │     ├─ 合并去重 → 候选 Skill 列表（最多 10 条）
  │     │
  │     └─ LLM 相关性校验 (SKILL_RELEVANCE_VERIFY_PROMPT)
  │         输入：本次审查的招标文件元信息 + 候选 Skill
  │         输出：每条 Skill 的 relevance_score + reason
  │         过滤：relevance_score >= 0.6
  │         排序：maturity × confidence × relevance DESC
  │         截断：top 3
  │
  ├─ _build_system_prompt(rule_doc_content, experience_skills)
  │     │
  │     └─ 在 SYSTEM_PROMPT_WITH_RULE 中填充 {experience_guidance}
  │
  └─ 进入正常 ReAct 循环
```

### 7.2 注入位置

## 十、人工反馈机制（Phase 3 增强设计）

> 本章节是对 Phase 3「用户反馈闭环」的完整增强设计，补充了前端交互、审批管线、反馈到经验的完整流程、经验仪表盘。

### 10.1 为什么人工反馈是关键拐点

人工反馈是系统从"自嗨式反思"跃升到"人机协同学习"的关键，理由：

1. **质量评分校准**：`quality_score = 0.6 × QualityEvaluator + 0.4 × LLM` 两者都是自动化系统，存在系统性偏差。人工反馈提供领域专家的 ground truth。
2. **负学习防护**：仅靠自动评分无法可靠检测错误——用户说"这条发现是错的"比任何自动评分都更可信。
3. **不对称更新验证**：confirm +0.05 vs contradict -0.2 的规则需要真实用户行为数据来验证参数是否合理。
4. **冷启动加速**：专家用户早期反馈可快速播种高质量 Skill。
5. **领域知识注入**：用户修正能注入规则文档中不存在的隐性知识。

### 10.2 `experience_feedback` 表增强

在第四章 4.5 节基础上扩展为 16 字段：

```sql
CREATE TABLE experience_feedback (
    id              VARCHAR(36) PRIMARY KEY DEFAULT gen_random_uuid(),
    finding_id      VARCHAR(36) NOT NULL REFERENCES review_results(id) ON DELETE CASCADE,
    user_id         VARCHAR(36) NOT NULL REFERENCES users(id),
    project_id      VARCHAR(36) NOT NULL REFERENCES projects(id),
    task_id         VARCHAR(36) NOT NULL REFERENCES review_tasks(id),

    feedback_type   VARCHAR(20) NOT NULL CHECK (feedback_type IN ('confirm', 'contradict', 'refine')),

    -- 反对专用
    contradict_reason VARCHAR(50),  -- should_comply / severity_too_high / severity_too_low / item_not_applicable

    -- 修正专用
    corrected_severity   VARCHAR(50),
    corrected_suggestion TEXT,
    corrected_is_compliant BOOLEAN,

    -- 通用
    comment         TEXT,

    -- 审批管线
    status          VARCHAR(20) NOT NULL DEFAULT 'pending'
                    CHECK (status IN ('pending', 'accepted', 'rejected', 'superseded')),
    reviewed_by     VARCHAR(36) REFERENCES users(id),
    reviewed_at     TIMESTAMPTZ,

    -- 经验联动
    affected_skill_id   VARCHAR(36),
    confidence_delta    REAL NOT NULL DEFAULT 0.0,

    -- 批量追踪
    batch_id        VARCHAR(36),
    rule_doc_name   VARCHAR(255),

    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**多用户冲突解决**：同一 finding 可能收到不同用户的矛盾反馈。内部用户（`interior_user=true`）权重 2x，外部用户 1x，加权多数决定管线动作。

### 10.4 反馈处理管线（五阶段）

```
用户提交反馈
     │
     ▼
[1] 即时：验证 + 持久化 (status=pending)     ← API 同步处理
     │
     ▼
[2] 异步 Celery：解析技能关联 (affected_skill_id)
     │
     ▼
[3] 异步：自动接受 或 排队等待人工审批
     │
     ├── 自动接受 ──→ [4] 立即应用置信度变更
     │
     └── 等待审批 ──→ 内部用户审批 ──→ [4] 应用变更
     │
     ▼
[4] 应用置信度变更到 Skill
     │
     ▼
[5] 触发下游：Skill 重写（refine）或退休（contradict 低于阈值）
```

#### 自动接受条件

| 反馈类型 | 自动接受条件 |
|---------|------------|
| confirm | 任何认证用户（直接接受） |
| contradict | ≥2 用户反对同一条，或反对者为内部用户 |
| refine | 修正值在合法范围内，且不改变 is_compliant（从 false→true 需审批） |

## 十一、风险评估与缓解

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| LLM 自评不准确，产生错误经验 | 中 | 中 | 双源融合（QualityEvaluator × 0.6 + LLM × 0.4）+ 成熟度门控 + 用户反馈校准 |
| 经验注入占用过多 Context 窗口 | 低 | 中 | 严格限制注入量（3 条/每次，≤600 tokens） |
| 冷启动期效果不明显 | 高 | 低 | 规则文档正例/反例可手工播种为初始 Skill |
| 模型升级导致经验失效 | 低 | 中 | `experience_cases.metadata` 含模型版本，升级后可按版本批量重置 |
| Skill 提取的 LLM 输出格式漂移 | 中 | 中 | 严格 JSON Schema 校验 + 失败重试 + 兜底正则解析 |
| 跨规则文档污染 | 低 | 高 | `group_id` 硬隔离 + Cluster LLM 域校验硬约束禁跨 group 合并 |
| EverOS 上游 Prompt 漂移 | 中 | 低 | Prompt 锁定到具体 commit；定期同步上游；保留 diff |
| EverOS 依赖锁定 | 低 | 中 | EverOS 作为 third_party 子模块，不直接 import；Prompt 与数据模型为概念参考，代码自实现 |
| 反思不够准确导致反复修正同一问题 | 中 | 低 | 假设晋升 + 矛盾案例 −0.2 不对称更新，让错误经验快速退休 |
